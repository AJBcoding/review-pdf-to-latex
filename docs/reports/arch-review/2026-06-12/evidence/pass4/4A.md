### Doc-open pipeline is four cloned loaders (~450 LOC) whose hand-maintained reset/dispose lists have already diverged   [severity: high] [effort: M]
- FACT (cited): `loadPdf` (renderer/index.ts:1230–1377), `loadMarkdown` (1379–1507), `loadHtml` (1509–1601), `loadDocx` (1603–1681) each repeat the same sequence: drain md-save debounce, unsubscribe file watcher, drain drafts debounce, a ~13-line manual docState/UI reset (1257–1277 vs 1398–1412 vs 1527–1542 vs 1614–1629), `readPdfBytes`, `loadDraftsForCurrentDoc`, viewer construction, an identical agent-notify tail including four copies of the inline `window as unknown as { agentViewer?: … }` cast (1344–1357, 1485–1503, 1579–1597, 1665–1677), then `fileTree?.setActiveFile` + `scheduleAppStateSave()` + the `toolbar:doc-state-changed` event (1371–1376, 1504–1506, 1598–1600, 1678–1680). The lists have already drifted: loadPdf disposes all three alt viewers (1240–1245); loadMarkdown disposes ONLY `mdViewerRef` (1426–1429) — `htmlViewerRef`/`docxViewerRef` survive an html/docx→md switch; loadHtml disposes md+html but not docx (1519–1520); only loadDocx disposes all three (1609–1611). Selection caches likewise: `lastMdSelection` is reset only in loadMarkdown (1431); `lastHtmlSelection` only in loadHtml/loadDocx (1537, 1624); loadPdf resets neither (grep over `lastMdSelection|lastHtmlSelection`, assignment sites 172–173, 1431, 1442, 1537, 1560, 1624, 1646 — none inside loadPdf).
- WHY IT MATTERS: every new format means a fifth clone and a fifth hand-copied reset list; the dispose/selection asymmetries are live drift evidence (leaked viewer instances with listeners; staleness is currently masked only because `updateAnchorMeta`/`handleSubmit` branch per format at 2084–2110 and 2159–2175).
- RECOMMENDATION: extract one `openDocument(path)` = teardownCurrent() + resetDocSession() + a per-format adapter `{ createViewer, load, postLoad }` selected from a registry keyed by `classifyPath`; the four loaders shrink to adapter entries. M. This registry is also exactly the seam the multi-format capability direction needs.

### Concurrent doc opens race: docState mutated across awaits with no epoch guard — can pair doc B's path with doc A's sha256 and write A's comments under B's sidecar   [severity: high] [effort: S–M]
- FACT (cited): `loadMarkdown` sets `docState.path = path` (renderer/index.ts:1398), then awaits `readPdfBytes` (1415), then sets `docState.sha256 = bytesResult.sha256` (1423) with no re-check that `docState.path` still equals its `path` argument. `loadPdf` has the same shape (1257, await at 1284–1287, sha write at 1300); its stale-doc check at 1311–1314 runs AFTER the 1300 sha assignment and its mismatch branch is an empty comment — "continue with viewer load below; … nothing to do here." — so a stale loader still calls `await h.viewer.loadBytes(bytesResult.bytes)` (1327) into the shared viewer. Only `loadDraftsForCurrentDoc` carries a real stale guard (`if (docState.path !== path || docState.sha256 !== sha256) return;`, 1817). All entry points funnel here: tree click, palette pick, external open, and app-state restore all call `openFileFromTreeOrPalette` (608, 614, 630, 875) with no serialization. The await window spans a full-file IPC byte read (readPdfBytes is the app's generic loader, 1415/1545/1632), so it widens with document size.
- WHY IT MATTERS: two quick opens (tree double-click on different rows; palette Enter then another) interleave so the second loader resets `docState.path` to B while the first resumes and stamps `docState.sha256` with A's hash; subsequent `flushDraftsWrite` (229–239, reads `docState.path`/`sha256`/`comments` at flush time) persists the mismatched pair — silent drafts mis-keying in the data model the charter says matters most.
- RECOMMENDATION: module-level load epoch (`const my = ++loadEpoch`, re-check after every await, abort on mismatch) or a single-flight open queue inside `openFileFromTreeOrPalette`. Make loadPdf's decorative guard real in the same change. S–M.

### Skip-permissions setting is read through a window-global that the new-agent-pane branch never wires — flushAppStateSave silently rewrites it to `true`   [severity: med] [effort: S]
- FACT (cited): `bootClaudeSettings()` is called only on the legacy branch of `bootClaudePane` (renderer/index.ts:449); the new-pane branch returns at 432 (after `bootToolbar()` at 431) without calling it. bootClaudeSettings exposes the checkbox via a window global: `(window as any).__claudeSkipPermissions = () => skipCheck?.checked ?? true;` (487). `flushAppStateSave` persists `claude_dangerous_skip_permissions: (window as any).__claudeSkipPermissions?.() ?? true` (804) and is debounce-triggered by tree/layout/doc activity (e.g. 1372, 642, 651, 730). `grep -rn '__claudeSkipPermissions' desktop/` hits only renderer/index.ts:487 and :804 — no other writer.
- WHY IT MATTERS: with the new agent pane active, the global is undefined, so every app-state write resets the persisted flag to `true`; a user who explicitly disabled `--dangerously-skip-permissions` on the legacy pane (the toggle shipped in commit 8a98733) gets the dangerous default silently restored after flipping the pane flag and back. Mitigation: `true` is also the field's default (read site 467), so only explicit opt-outs are harmed — hence med, not high. Root cause is a window global used as a cross-module settings API.
- RECOMMENDATION: hold the setting in module state (seed from `readAppState`, update on the change listener), have flushAppStateSave read that; and make flushAppStateSave spread the previously-read state so fields with no live UI can never be clobbered to defaults. S.

### draftsCache: path-keyed, never invalidated, ignores doc_version, and aliases the live comments array   [severity: med] [effort: S]
- FACT (cited): `draftsCache` (renderer/index.ts:217) has only set sites — 238 (`flushDraftsWrite`, storing a `DraftsFile` whose `comments` is the live `docState.comments` reference, 232–238) and 1831 (`loadDraftsForCurrentDoc`); grep over index.ts shows no `.delete`/`.clear`. The cache-hit path (1809–1815) keys on path alone, ignores the current sha256, and re-aliases `docState.comments = cached.comments` (1812). Meanwhile `seedNextVersionDraft` writes a fresh drafts file straight to disk via `writeDrafts(newDocId, …)` (2959) without touching the cache.
- WHY IT MATTERS: reopening a doc that was cached earlier in the session serves stale comments even if the drafts file changed on disk — and the seed-then-reopen sequence is precisely the iterative round loop the tool exists for: the cached (pre-seed) array masks the seeded v1.1 comments, and the next debounced flush can overwrite the seeded file. (seedNextVersionDraft's logic is lane 4C's; the cache lifecycle is structural and cited here.)
- RECOMMENDATION: key the cache by `path + sha256` and invalidate on any out-of-band write — or simply delete it: `readDrafts` is one IPC file read per doc open; the cache buys almost nothing. S.

### Three hand-rolled debounce instances plus eight inline cancel-and-flush drains   [severity: med] [effort: S]
- FACT (cited): the identical timer/schedule/flush pattern is implemented three times — md save (renderer/index.ts:176–206, 500 ms), drafts (215–256, 250 ms), app state (568–569, 782–810, 250 ms) — and the "cancel timer then flush" drain is re-inlined at wireDraftsQuitFlush (900–909), beforeunload (917–927), writeBundle (1168–1172), loadPdf (1232–1236, 1249–1253), loadMarkdown (1383–1392), loadHtml (1512–1516, 1521–1525), and loadDocx (1606, 1612). The quit handshake had to be retrofitted for the md timer (comment "Flush both drafts and md source saves", 898).
- WHY IT MATTERS: data-not-lost-on-quit correctness depends on every drain site knowing every timer; the next persisted surface (docx/html comments) must be hand-added at four-plus sites or it silently misses the quit flush.
- RECOMMENDATION: a tiny `Debounced` utility (`schedule/flush/cancel/pending`) plus a registry array the quit-flush and loaders iterate. S.

### Per-operation cost: every debounced md save re-reads the whole file over IPC just to recompute its sha256; first tree-filter keystroke walks the entire root   [severity: low] [effort: S]
- FACT (cited): `flushMdSave` writes the buffer (renderer/index.ts:197) then calls `rehashCurrentDoc`, which round-trips the FULL file bytes through `readPdfBytes` solely to read `res.sha256` (208–212) — on every 500 ms save flush while typing. Separately, tree.ts `setFilter` calls `loadAllForFilter` on every keystroke of a non-empty query (tree.ts:267–270), which stack-walks and `listDir`s every directory under the root (tree.ts:276–290), defeating the lazy-load rationale in the module's own header ("opening a large repo (or accidentally pointing at $HOME) stays snappy — we only read the directories the user has actually opened", tree.ts:3–5). Subsequent walks are cache-served (ensureLoaded, 203–218) but the first is unbounded.
- WHY IT MATTERS: IO churn scales with document/tree size in the two highest-frequency interactions (typing in the editor; typing in the filter).
- RECOMMENDATION: have `fs:writeFileText` return the new sha256 (main already holds the bytes); debounce the filter walk and/or cap depth with a "load more" affordance. S.
