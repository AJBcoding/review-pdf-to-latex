### Every failure-recovery affordance in the submit flow is a dead end or violates its own stated contract   [severity: high] [effort: M]

- FACT (cited): Three independent recovery paths are wired but broken. (a) **Timeout "Re-sling" is a guaranteed no-op**: the timeout banner's Re-sling button calls `retrySling()` (submit.ts:424), which dispatches `submit:retry-requested` (submit.ts:453-455); index.ts's only listener re-runs `handleSubmitBundle()` (index.ts:362-364), whose first guard is `if (!submitCanFire() || submitIsInFlight())` ‚Üí flash "Submit already in flight." (index.ts:1093-1095) ‚Äî and in the `timeout` state `canFire()` is false (`timeout` absent from submit.ts:716-721) and `isInFlight()` is true (`timeout` included, submit.ts:136-140). So clicking Re-sling after a timeout always dead-ends with a contradictory message. (b) **"Resume round" is a stub that points at a blocked path**: the interrupted-round banner's Resume button only flashes "Press Cmd+Return to re-sling this round." (index.ts:2778-2783), but Cmd+Return ‚Üí `handleSubmitBundle` is blocked by `hasInProgressRound()` ‚Üí "A round is already in progress for this doc ‚Äî resume or abandon it first." (index.ts:1097-1100; `hasInProgressRound` returns true for any `round_status === 'in_progress'` round, index.ts:1143-1148 ‚Äî and "interrupted" is defined as a subtype of in_progress, index.ts:2878-2887). Resume tells the user to do the one thing the app refuses; the only working exit is Abandon. (c) **Retry mints a new round while claiming it re-sends the same one**: `retrySling`'s own comment says "Retry re-sends the same payload (same submit_id) so the rig sees a single round" (submit.ts:439-441) and the timeout banner says "Re-sling resends the same payload (same submit_id, no new round file)" (submit.ts:417) ‚Äî but the retry path re-executes `handleSubmitBundle` ‚Üí `executeSubmit` ‚Üí `submitPromote` (index.ts:362-364, 1105-1137; submit.ts:210-219), and `promoteDraft` mints a fresh `submitId` and a fresh `submit-<id>.json` on every call (main/submit.ts:88-92). The event detail carrying `{submitId, destination}` (submit.ts:453-455) is ignored by the listener, so a no-origin user is also re-prompted by the destination picker (submit.ts:199-206).
- WHY IT MATTERS: The happy path works; every unhappy path strands the user. Timeout recovery is dead, interrupted-round resume is circular, and "retry" silently violates the rig-side dedup contract by creating duplicate frozen rounds whose results pairing is keyed on submit_id alone.
- RECOMMENDATION: Give the submit module a real retry entry point: `retrySling(submitId, destination)` that calls `submitSling` directly against the existing `promote.submitFilePath` (cache it next to `pendingSubmitId`), skipping promote and the picker; allow `canFire()`/`handleSubmitBundle` to accept the retry path in `timeout`/`send_failed`; wire Resume to that same entry point instead of the flashHint stub. M ‚Äî the state machine and IPC calls all exist; this is re-plumbing, not new machinery.

### Re-opening a doc with a completed round re-seeds the next version's draft, clobbering any comments added there since ‚Äî latent data loss   [severity: high] [effort: S]

- FACT (cited): `seedNextVersionDraft` writes the v1.1 drafts file wholesale ‚Äî `const file: DraftsFile = { schema_version: 1, doc_version: newSha, comments: reraised }` then `writeDrafts(newDocId, newSha, file)` (index.ts:2954-2959) ‚Äî where `reraised` contains ONLY the deferred/needs-followup re-raises (index.ts:2929-2952). The only guard against re-running is the in-memory `seeded` flag (index.ts:2674-2679), which (a) defaults to false whenever the round isn't already in the map (`seeded: prev?.seeded ?? false`, index.ts:2660-2667) and (b) lives in `docState.rounds`, which is rebuilt to `new Map()` on every doc load (index.ts:1261, 1402, 1531, 1618). The results watcher re-emits every results file on each doc open (initial scan ‚Äî REVIEW.md Pass-3 finding, results-watcher.ts:152-158), so re-opening the v1 doc re-fires the complete-round event with `seeded` false and the seed re-runs. The in-code comment misjudges the consequence: "The existing draft would be overwritten cleanly (same sha256 path) but the dup work is wasted I/O" (index.ts:2964-2966).
- WHY IT MATTERS: Real workflow: round completes ‚Üí user opens v1.1, writes new comments (persisted to the path-keyed sidecar) ‚Üí later re-opens v1 to compare ‚Üí seed re-fires and the v1.1 sidecar is atomically replaced with only the re-raises. The user's v1.1 comments are gone with no error, no prompt, and no backup. This is the drafts-sidecar equivalent of the engine's apply-before-validate corruption class.
- RECOMMENDATION: Make seeding idempotent against disk, not memory: before writing, `readDrafts(newDocId)` and skip (or merge re-raises into) a non-empty existing draft; alternatively persist a `seeded_from: <submit_id>` marker in the seeded DraftsFile and check it. Either is S and testable without the rig.
