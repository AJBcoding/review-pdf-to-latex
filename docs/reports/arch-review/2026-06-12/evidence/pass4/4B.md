### Saving an edited Markdown file silently deletes its frontmatter   [severity: high] [effort: S]

- FACT (cited): `MarkdownViewer.loadBytes` strips frontmatter before mounting the editor — `const { frontmatter, body, bodyOffset } = parseFrontmatter(text); ... this.mountEditor(body);` (md-viewer.ts:118-124) — and `getContent()` returns only the editor doc: `return this.editorView?.state.doc.toString() ?? '';` (md-viewer.ts:146-148). The save path writes that body-only string straight to the source file: `flushMdSave` does `const content = mdViewerRef.getContent();` then `await window.electronAPI.writeFileText(docState.path, content)` (renderer/index.ts:192-197), debounced 500 ms from `onContentChange` (renderer/index.ts:1437-1439) and flushed on blur (renderer/index.ts:1452-1457). Nothing reattaches the stripped block: `grep -n 'frontmatter' renderer/index.ts` → no hits, and the viewer instance does not retain the raw frontmatter text — only `bodyOffset`, which is written and never read anywhere (`grep -rn 'bodyOffset'` → md-viewer.ts:95, :121, plus the parseFrontmatter internals :281, :284, :296 — zero read sites).
- WHY IT MATTERS: The first keystroke in any frontmatter'd .md file permanently deletes its YAML block from disk — data loss in the app's ONLY editable format, on the daily-driver path. It also breaks the sidecar's own rename-recovery input: `DocFingerprint.title_from_frontmatter` (shared/types.ts:176) is computed from content this save path destroys.
- RECOMMENDATION: Store the raw frontmatter match on the instance in `loadBytes` and prepend it in `getContent()` (or mount the full text and decorate/fold the frontmatter region instead of excising it — `bodyOffset` exists precisely for that translation and is currently dead). Add a vitest: load doc-with-frontmatter → edit → getContent contains the block. S.

### HTML/DOCX anchors are smuggled through `md_anchor` with `as any` — the renderer half of the anchor-model flaw, plus a misreporting discriminator   [severity: high] [effort: M]

- FACT (cited): `buildHtmlCommentPayload` persists a hybrid object into the typed `md_anchor` field with extension fields the schema doesn't have, cast away: `md_anchor: { char_start: sel.charOffset, ..., quoted_text: sel.text, selector: sel.selector, char_offset: sel.charOffset, char_length: sel.charLength } as any` (renderer/index.ts:2214-2223; function at :2198), with `prefix: ''`/`suffix: ''` degenerate context and the same value duplicated under two names (`char_start` :2215 = `char_offset` :2221). Read-back goes through untyped sniffing: `.filter((c) => c.md_anchor && (c.md_anchor as any).selector)` (renderer/index.ts:1686, duplicated verbatim at :1702). The declared discriminator has only two values — `export type AnchorKind = 'pdf-glyph-rect' | 'md-fuzzy-snippet';` (shared/types.ts:170) — and both iframe viewers report the md kind for selector anchors: `get anchorKind(): AnchorKind { return 'md-fuzzy-snippet'; }` (html-viewer.ts:55; docx-viewer.ts:41), while their actual payload is the CSS-selector `HtmlAnchor`/`HtmlSelection` shape (html-viewer.ts:4-16). Every html/docx comment also carries the required placeholder PDF anchor `anchor: { page: 1, region: { x: 0, y: 0, w: 0, h: 0 } }` (renderer/index.ts:2204, same at :2237 for md) — the exact value 3C showed `bundle.ts` consumes blind (REVIEW.md "Bundle writer is anchor-kind-blind"). Separately, the md anchor shape itself is defined twice with no link: `MdAnchor` (shared/md/anchors.ts:1-7) vs the inline `md_anchor` literal (shared/types.ts:157-165).
- WHY IT MATTERS: There are FOUR real anchor shapes (pdf AnchorRegion, MdAnchor, HtmlAnchor, and the hybrid above) behind TWO declared kinds, one of which lies. Sidecars on disk now contain fields no schema describes; any consumer trusting `anchor_kind === 'md-fuzzy-snippet'` to mean MdAnchor (the engine, the rig, or `fuzzyMatchAnchor` itself if html docs ever hit the md path) gets degenerate prefix/suffix anchors; and the zero-rect placeholder is one call-site mistake away from a nonsense PDF Highlight. This is the live renderer-side proof for the discriminated-union restructure 3A/3C named as the highest-leverage pre-work for the unified model.
- RECOMMENDATION: Add the html/selector kind to the per-comment discriminated anchor union when the 3A/3C restructure lands (`{ kind: 'pdf-glyph-rect', ... } | { kind: 'md-fuzzy-snippet', ... } | { kind: 'html-selector', ... }`), make `FileViewer.anchorKind` report it truthfully, delete the `as any` write/read sites, and make `MdAnchor` the single definition imported by types.ts. M (rides the already-recommended restructure; sidecar shim required).

### html-viewer and docx-viewer are ~60% verbatim copies of each other — and the copies have already drifted on wrong-text fallback behavior   [severity: med] [effort: S–M]

- FACT (cited): `cssPath` is byte-identical in both files (html-viewer.ts:172-195 vs docx-viewer.ts:177-200); `injectStyles` injects the same `.review-highlight` + `::selection` CSS (html-viewer.ts:91-107 vs docx-viewer.ts:101-114); `wireSelectionCapture` is the same mouseup→rAF→selector/charOffset capture (html-viewer.ts:109-138 vs docx-viewer.ts:116-141); constructor/stage/iframe setup is identical including the docx viewer reusing the html viewer's CSS class names `html-viewer-stage`/`html-viewer-iframe` (docx-viewer.ts:21-26 vs html-viewer.ts:35-39). The drift: on a failed text match, html-viewer's `findTextNode` falls back to returning the FIRST non-empty text node under the element (html-viewer.ts:203-208 — comment "Fallback: any text node with partial match", but the predicate is `textContent.length > 0`), so `applyHighlights` then applies `char_offset` to an unrelated node and highlights wrong text; docx-viewer's inline walker has no such fallback and skips the anchor instead (docx-viewer.ts:157-171). The host-side consumers are twins too: `applyDocxCommentHighlights` (renderer/index.ts:1683-1697) and `applyHtmlCommentHighlights` (:1699-1713) are identical bodies.
- WHY IT MATTERS: Every upcoming html/docx feature in the charter direction (comment UI, comments.xml round-trip display) must currently be built and bug-fixed twice; the fallback divergence shows drift is already producing user-visible behavior differences (silently-wrong highlight vs silently-missing highlight).
- RECOMMENDATION: Extract one `IframeDocViewer` base (stage/iframe/sandbox, injectStyles, wireSelectionCapture, applyHighlights, cssPath, findTextNode) with a `bytesToHtml(bytes): Promise<string>` strategy — html = decode+base-href injection, docx = mammoth — and collapse the two index.ts highlight functions into one. Pick ONE not-found behavior (skip, like docx). S–M; do it before the docx/html comment-UI epics (rev-2h6, rev-6k6).

### Selector/charOffset anchor capture is structurally lossy: first-occurrence offsets, −1→0 coercion, single-node assumption, silent surroundContents failure   [severity: med] [effort: M]

- FACT (cited): Capture computes the offset as an indexOf over the parent element's text: `const nodeText = container?.textContent ?? ''; const charOffset = nodeText.indexOf(text);` then `charOffset: Math.max(0, charOffset)` (html-viewer.ts:127-133; docx-viewer.ts:131-137) — first occurrence wins for repeated snippets, and a selection spanning multiple elements (where `text` exceeds `container.textContent`) yields −1, coerced to 0. `container` itself is only `range.startContainer.parentElement` (html-viewer.ts:125; docx-viewer.ts:129), so multi-block selections record only the start element. Resolution then applies that element-relative offset to a single TEXT NODE: `range.setStart(textNode, start); range.setEnd(textNode, end)` with `end = Math.min(textNode.length, ...)` (html-viewer.ts:157-164; docx-viewer.ts:161-168) — offsets computed against `element.textContent` are applied to one child text node, a different coordinate space whenever the element has multiple children — and `range.surroundContents(mark)` throws on any range that partially selects a non-text node, swallowed by the bare `catch { /* skip silently (orphaned) */ }` (html-viewer.ts:164-167; docx-viewer.ts:166-172).
- WHY IT MATTERS: For exactly the selections reviewers make (multi-paragraph, spanning bold/links), the persisted anchor is wrong-by-construction and the failure is invisible (no orphan state surfaces to the card UI — contrast the md path, which tracks `orphaned` explicitly, md-viewer.ts:33-57). JUDGEMENT: for the charter's DOCX comments.xml round-trip these anchors are also the wrong substrate — they are mammoth-output-DOM-relative (`nth-of-type` paths against regenerated HTML), with no stable mapping back to OOXML ranges.
- RECOMMENDATION: Reuse the md strategy for iframe formats: capture `prefix/quoted_text/suffix` context against the iframe's extracted linear text (the `createMdAnchor` function is already doc-agnostic over strings) and resolve with `fuzzyMatchAnchor` + a Range built by walking text nodes — keep the CSS selector only as a locality hint. This both fixes the lossiness and gives DOCX an anchor that can survive re-conversion. M; design it inside the unified anchor union, not before it.

### The FileViewer interface is decorative: never used as a type, PDF-shaped, and missing everything the viewers actually share   [severity: med] [effort: M]

- FACT (cited): `grep -rn 'FileViewer' renderer/ shared/ --include='*.ts'` → outside its own file, every hit is an `implements FileViewer` clause or type import in the four viewers (pdf-viewer.ts:78, md-viewer.ts:88, html-viewer.ts:24, docx-viewer.ts:11); the host never holds one — it keeps four parallel concrete refs `viewerRef: PdfViewer | null` / `mdViewerRef` / `htmlViewerRef` / `docxViewerRef` (renderer/index.ts:168-171). The interface's surface (shared/file-viewer.ts:8-21) is paging/zoom — `nextPage/prevPage/fitPage/fitWidth` — which three of four viewers stub as empty no-ops (md-viewer.ts:126-129; html-viewer.ts:69-72; docx-viewer.ts:79-82), while everything the app actually needs polymorphically is absent: selection callbacks (four different shapes: `SelectionPayload` pdf-viewer.ts:54-62, `MdSelection` md-viewer.ts:17-21, `HtmlSelection` html-viewer.ts:11-16, and an inline literal docx-viewer.ts:8), anchor highlighting (`revealAnchor` pdf-viewer.ts:288, `setTrackedAnchors` md-viewer.ts:161, `setHighlightedAnchors` html-viewer.ts:86/docx-viewer.ts:96), and load context (HtmlViewer alone needs `basePath`, html-viewer.ts:20). `FileViewerPageInfo` (shared/file-viewer.ts:3-6) has zero consumers and pdf-viewer re-declares it inline (`onPageInfo?(info: { page: number; totalPages: number })`, pdf-viewer.ts:75). Selection-clear semantics also diverge: pdf never emits a null/cleared event (`captureSelection` early-returns on collapse, pdf-viewer.ts:397), md/html/docx all emit `null` (md-viewer.ts:204-205; html-viewer.ts:115-117; docx-viewer.ts:122-125).
- WHY IT MATTERS: The abstraction exists but buys nothing — the host's per-format if/else dispatch (4A's finding) is forced partly because the interface can't carry the operations the host needs, so each new format adds a fourth nullable global and another dispatch arm. This is the renderer-side seam the unified comment model has to land on.
- RECOMMENDATION: Renegotiate FileViewer around the anchor union: `loadBytes(bytes, ctx)`, `onSelection(unifiedSelection | null)`, `applyAnchors(Anchor[])`, `reveal(Anchor)`, plus a `capabilities` record (paged, editable, zoomable) so chrome enables controls by capability instead of format checks; host holds ONE `FileViewer` ref. M — pairs with (and partly pays for) the 4A dispatch-registry work.

### Fuzzy re-anchoring persists unverified guesses back into the sidecar, destroying the user's original referent   [severity: med] [effort: S–M]

- FACT (cited): `fuzzyMatchAnchor` steps 3-4 return matches WITHOUT verifying the text at the guessed position: the prefix path computes `expectedStart/expectedEnd` and returns `confidence: 'fuzzy'` with no comparison of `doc.slice(expectedStart, expectedEnd)` against `quoted_text` (shared/md/anchors.ts:54-63, suffix variant :65-74, 20-char partial :77-87), and every step uses `doc.indexOf(...)` — first occurrence — so a repeated snippet/prefix anywhere earlier in the doc captures the anchor (:44, :55, :67, :79). The host then PERSISTS the guess: `reanchorMdComments` writes `c.md_anchor.char_start = match.from; c.md_anchor.char_end = match.to;` (renderer/index.ts:1758-1759, function :1745), and on every content change `syncMdAnchorsToComments` rewrites the provenance fields from current positions — `c.md_anchor.quoted_text = doc.slice(a.from, a.to); c.md_anchor.prefix = doc.slice(Math.max(0, a.from - 40), a.from); ...` (renderer/index.ts:1775-1781, function :1764; note the `40` re-hardcodes `CONTEXT_CHARS` from anchors.ts:9). anchors.test.ts covers 5 happy paths only — no repeated-snippet case, no suffix-path case, no partial-path case, no false-positive guard case (anchors.test.ts:31-74). [Cross-lane: the two write-back functions are 4C's index.ts spans; the resolution semantics and test gap are this lane's.]
- WHY IT MATTERS: One wrong fuzzy hop is permanent — the next sync rewrites `quoted_text/prefix/suffix` to describe the WRONG location, so the original highlight (the reviewer's actual referent, and what a rig redraft targets) is unrecoverable from the sidecar. This anchor algorithm is also the designated reuse substrate for DOCX/HTML text anchoring (recommendation above), so its robustness is leveraged.
- RECOMMENDATION: In steps 3-4, verify the candidate slice (similarity threshold against `quoted_text`; downgrade to `orphaned` on failure); prefer the occurrence nearest `char_start` when `indexOf` is ambiguous (scan all occurrences). Keep the ORIGINAL quoted_text/prefix/suffix immutable and write relocations to separate fields (the schema already has the `new_anchor` precedent for PDF, shared/types.ts:150-151). Export CONTEXT_CHARS. Extend anchors.test.ts with the four missing cases. S–M.

### PdfViewer.renderPage is not serialized — concurrent renders share one canvas and mutable page state   [severity: low] [effort: S]

- FACT (cited): `renderPage` is async, awaits `this.doc.getPage(...)` then `this.page.render({ canvasContext: ctx, viewport: this.viewport, canvas: this.canvas }).promise` against instance-shared `this.canvas`/`this.viewport`/`this.page` (pdf-viewer.ts:166-243, render call :192-196), with no in-flight guard, queue, or RenderTask cancellation anywhere in the file (the only cancellation is the TEXT layer's, `cancelTextLayer` pdf-viewer.ts:248-265). `nextPage`/`prevPage`/`gotoPage`/`setZoom` each call `renderPage` directly (pdf-viewer.ts:268-282, :335-341).
- WHY IT MATTERS: Two overlapping calls (rapid paging, zoom during page render) interleave canvas resizes and render passes; pdf.js rejects concurrent `render()` on one canvas, so the result is an unhandled rejection or a page painted with another page's viewport state. Low because single-user paging usually awaits, but it is the kind of race that surfaces as "sometimes the page is blank".
- RECOMMENDATION: Keep the current `RenderTask`, cancel it at the top of `renderPage` (mirroring `cancelTextLayer`), and/or chain calls through a `lastRender: Promise` queue. S.

### DOCX dark-mode semantics are inverted: hardcoded dark document + shared invert filter   [severity: low] [effort: S]

- FACT (cited): docx-viewer hardcodes a dark document inside the iframe — `color: #e6e6e6; background: #222;` (docx-viewer.ts:59-60) — while `setDarkMode` toggles the same `html-dark` class as the html viewer (docx-viewer.ts:84-87), whose CSS is an invert filter: `.html-viewer-stage.html-dark { filter: invert(0.88) hue-rotate(180deg); }` (renderer/styles.css:1828-1830).
- WHY IT MATTERS: With dark mode OFF the DOCX page renders dark (ignoring the light theme); turning dark mode ON inverts the already-dark page to light — exactly backwards relative to every other viewer.
- RECOMMENDATION: Make the docx body styles theme-neutral (light, like html docs) so the shared invert filter behaves identically for both iframe viewers — one CSS block change, naturally folded into the IframeDocViewer extraction above. S.

**Also noted (cited):** `md-viewer.ts:15` imports `type { MdAnchor }` that nothing in the file references (grep: no other `MdAnchor` occurrence in md-viewer.ts) — residue that strict `noUnusedLocals` would catch if the typecheck ran (3A found the first `npm run typecheck` invocation is a no-op). The frontmatter regex `/^---\s*\n([\s\S]*?)\n---\s*\n?/` (md-viewer.ts:283) lazily matches the FIRST closing `---`, so an md file opening with a horizontal-rule-delimited block is silently eaten as frontmatter (compounds the data-loss finding). html-viewer's base-href fallback prepends `<head>` before any doctype when the doc lacks a head tag (html-viewer.ts:65) — malformed but browser-tolerated. A tracked md anchor orphaned by an edit stays orphaned even after undo restores the text (`if (a.orphaned) return a;`, md-viewer.ts:50-53) until a reload triggers `reanchorMdComments`. PdfViewer registers a document-level `mouseup` listener per instance (pdf-viewer.ts:141), removed only in `dispose` (:381) — correct today, but a leak the moment a host forgets dispose. Tests: all four viewer classes have zero tests (known per ground-truth map — assessed, not discovered); the cheap pure seams are `mergeRectsByLine` (pdf-viewer.ts:486-506), `cssPath`, `parseFrontmatter`, and the fuzzyMatchAnchor edge cases above. Out-of-scope, one line each: `rehashCurrentDoc` re-reads the whole file over IPC after every md save (renderer/index.ts:207-211) → 4A per-operation cost; `fs:readPdfBytes` naming for the generic byte loader already in 3A's tail.

**Reusability for the multi-format direction (JUDGEMENT, grounded above).** The most reusable asset in this lane is `shared/md/anchors.ts`: pure, dependency-free, string-typed (anchors.ts:11-15, :34) and already vitest-covered — with the verification guard added it is the text-anchor leg of the unified model for MD, HTML, and DOCX body text alike. The PDF leg is equally ready: `SelectionPayload` capture (pdf-viewer.ts:394-445), `revealAnchor`'s PDF-point→viewport projection (:288-317), and the persistent highlight overlay (:447-463) are exactly the display/navigation primitives an Acrobat-annotation layer needs; because the viewer already consumes pdf.js's reference builder chain (TextLayerBuilder + StructTreeLayerBuilder, pdf-viewer.ts:31-34, :204-240), adding the missing sibling (pdf.js's annotation layer, Pass 5's question) slots into `renderPage` rather than requiring a rework. CodeMirror's `anchorField` position-mapping through edits (md-viewer.ts:42-57) is the right live-tracking mechanism — keep it; it just needs the orphan-recovery and provenance fixes above. What does NOT carry forward: the selector/charOffset capture scheme (lossy by construction, finding above) and the two-copy iframe viewer pair — consolidate first, then build the html/docx comment UI once.
