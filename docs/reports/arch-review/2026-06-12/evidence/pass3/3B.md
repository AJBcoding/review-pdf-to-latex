### Two live Claude routes have materially divergent semantics, not just duplicate code   [severity: high] [effort: L]
- FACT (cited): The same four features are implemented twice with different behavior:
  1. **Permissions**: pty route defaults to `--dangerously-skip-permissions` (claude-pty.ts:219-220 `const skipPerms = params.dangerouslySkipPermissions !== false;` and again at :526-527); SDK route hardcodes `permissionMode: "default"` with an interactive `canUseTool` approve/deny flow (claude-backend.ts:161-162). The agent IPC payloads (`agent:send`, `agent:freshStart`, `agent:spawnSession` — agent-pane-ipc.ts:116-128, 216-228, 234-255) carry no skip-permissions field, so the Settings toggle (commit 8a98733) only affects the pty route.
  2. **Working directory**: pty route anchors cwd to the doc source dir per §9.2.9 (claude-pty.ts:209-210, 516-517); SDK route never sets cwd — `query()` options at claude-backend.ts:157-166 contain only `permissionMode`, `canUseTool`, `includePartialMessages`, `resume`, `model`. SDK sessions run in the Electron process cwd.
  3. **Worker caps & sling gating**: pty route enforces `MAX_WORKER_PTYS` = 16 (claude-pty.ts:494-500; shared/types.ts:942) and refuses sling without gt + destination (claude-pty.ts:503-511); `agent:spawnSession` has no session cap and no sling/gt validation (agent-pane-ipc.ts:234-255 validates only sessionId/prompt types).
  4. **Resume/continuity**: SDK conv session persists session.json and resumes (agent-pane-ipc.ts:48-54, session-store.ts:21-47); pty route has no resume at all — every spawn is a cold `claude` CLI start (claude-pty.ts:224-230).
- WHY IT MATTERS: Flipping the feature flag silently changes the security posture (skip-permissions vs approval prompts), the directory Claude operates in, resource limits, and continuity. For a daily-driver tool this is a behavioral fork, and every new agent feature must now be built and debugged twice (the Settings toggle already missed one route).
- RECOMMENDATION: Decide the convergence target (the SDK route is the structurally better base — typed events, native permission hooks, no TUI timing races) and write a short parity checklist (cwd, permissions param, worker cap, sling gating, resume) as bd issues; until the legacy pane is retired, route both paths through one shared "session policy" module so defaults can't diverge. L overall; the cwd and skip-permissions parity items alone are S each.

### Errored SDK session becomes a zombie: stays in the session map, silently eats messages   [severity: high] [effort: S]
- FACT (cited): When `query()` throws, claude-backend.ts:173-185 only emits a `session`/`error` event — it does not call `queue.end()`, does not resolve pending approvals, and nothing removes the session from the registry: `sessions.delete` happens only in `endSession` (agent-pane-ipc.ts:66-71). `ensureSession` returns any existing map entry unconditionally (agent-pane-ipc.ts:41-43), and `send()` just pushes onto the now consumer-less queue (claude-backend.ts:190-193).
- WHY IT MATTERS: After any backend error (network drop, SDK crash), every subsequent `agent:send` to that sessionId is accepted and silently dropped — the worst failure mode for a chat UI. Pending tool approvals also stay unresolved (the canUseTool promise leaks unless the SDK aborts it).
- RECOMMENDATION: In the `run()` catch (and after normal loop exit), end the queue, deny/clear `pending`, and notify the host to drop the map entry (e.g. an `onClosed` callback symmetric to `onSessionId`), so the next `agent:send` lazily creates a fresh session. S.

### agent:send lazily creates a session for ANY sessionId — unbounded, typo-tolerant spawn path   [severity: med] [effort: S]
- FACT (cited): `agent:send` calls `ensureSession({ sessionId: payload.sessionId, ... })` (agent-pane-ipc.ts:123-127), and `ensureSession` creates-if-missing for any id (agent-pane-ipc.ts:40-64). There is no counterpart to the pty route's `MAX_WORKER_PTYS` check (claude-pty.ts:494-500). So a send to a stale or mistyped worker id (workers are minted in the renderer as `worker-ctx-${Date.now()}`, renderer/toolbar.ts:342) silently starts a brand-new full Claude session.
- WHY IT MATTERS: Each session is a real `query()` subprocess/stream with cost; a renderer routing bug turns into N silent background sessions instead of an error. The pty route already treats "send to unknown worker" as a no-op (claude-pty.ts:658-661 returns when the handle is missing).
- RECOMMENDATION: Make `agent:send` create-only-for-conv: route worker creation exclusively through `agent:spawnSession`, make sends to unknown ids a logged no-op, and add a session-count cap mirroring MAX_WORKER_PTYS. S.

### Context-bundle priming serialized twice, in different layers, with different output formats   [severity: med] [effort: M]
- FACT (cited): The pty route serializes `ToolbarContextBundle` in MAIN — `bundleToPrimingText` + `buildCreateContextPriming` + `buildSlingPriming`, ~63 LOC emitting `# Context bundle` / `[Worker spawn — …]` shapes with β-marker instructions (claude-pty.ts:351-413). The agent route serializes the same bundle in the RENDERER — `bundleToPrompt` emitting a different `[Create Context]` / `Document:` shape with no marker grammar (renderer/toolbar.ts:214-237, cited as the other half; toolbar.ts itself is lane 4A's file). Doc-switch priming is split the same way: agent route builds `[Now reviewing: …]` in main (agent-pane-ipc.ts:92-98) while the pty route writes `[Now viewing: …]` from the renderer (renderer/claude-pane.ts:17-19 module comment).
- WHY IT MATTERS: The "rig-side priming language" that the worker/skill is primed with differs by which UI flag is set, so prompt-contract fixes (e.g. the truncation rules, the β grammar) must be made twice and can drift; the layering inconsistency (main vs renderer) also means there's no single seam to test priming text.
- RECOMMENDATION: Extract one pure, shared priming module (shared/ next to agent-pane/) that both routes call — bundle→text, doc-switch line, fresh-start line — with unit tests; it is also the natural place to make priming anchor-kind-aware for the multi-format direction (see capability note). M.

### claude-pty.ts mixes six concerns and its pure parser core has zero tests   [severity: med] [effort: M]
- FACT (cited): One 693-LOC file contains PATH binary discovery (`whichSync`, claude-pty.ts:89-105), the gt rig probe (:119-143), the TUI bracketed-paste protocol (:162-170), priming-text builders (:304-413), the β-marker line parser (`extractProgressMarkers`/`parseMarker`/`tokenizeMarker`, :419-488 — pure functions with no Electron/pty dependency), pty lifecycle for conv+workers, and IPC registration (:616-678). `find desktop/main -name '*.test.ts'` returns only `agent-pane-ipc.test.ts` and `sidecar-migration.test.ts` — no test touches claude-pty.ts, and the file itself notes its test seam has "no production caller" (claude-pty.ts:689-690).
- WHY IT MATTERS: The β-marker grammar is a hand-rolled protocol parsed by regex; it is exactly the kind of code that breaks on edge input (chunk boundaries mid-line are handled, but e.g. quoted-escape and CR handling are untested). It's also the file's most reusable piece and the cheapest to harden.
- RECOMMENDATION: Split the pure parts (marker parser, priming builders, whichSync) into importable modules with vitest coverage using the existing adapter.test.ts fixture style; leave pty lifecycle + IPC in place. M.

### Priming-by-setTimeout: two different magic-delay tuples racing the claude TUI   [severity: med] [effort: M]
- FACT (cited): Conversational priming fires at 1500ms then 350ms (claude-pty.ts:264-284); worker priming fires at 500ms then 400ms (claude-pty.ts:583-599). The 1500ms value is itself a bump from a lost race — the in-code comment records that 500ms "races that clear and the slash-command line disappears from visible scrollback" and pre-plans the *next* fix if it still fails (claude-pty.ts:258-263).
- WHY IT MATTERS: Startup-time-dependent priming is machine-speed-dependent and version-dependent (the comment pins behavior to "claude-code 2.1.146"); the two routes' inconsistent tuples mean any future race fix must be discovered twice. A slow machine or a claude startup change silently drops the skill activation.
- RECOMMENDATION: Trigger priming from observed output instead of wall-clock (wait for the first prompt-render data after spawn, or a quiet-period heuristic on `onData`), with the timeout kept only as fallback; unify conv/worker into one code path. M. (JUDGEMENT: this whole class of fragility disappears on the SDK route, which is an argument for the convergence finding above.)

### Capability note — the agent subsystem's context bundle is PDF-page-shaped, not anchor-shaped   [severity: med] [effort: M]
- FACT (cited): `ToolbarContextBundle` hardcodes page-centric context: `currentPage`, `pageCount`, `selection.page` + xywh region, `nearbyComments[].page`, and its doc comment says "Source PDF absolute path" (shared/types.ts:852-882). The priming serializer renders `p.${s.page}` (claude-pty.ts:365) and the doc-switch line guesses format from file extension to choose "editing/reviewing" and "file" vs "N pages" units (agent-pane-ipc.ts:94-97).
- WHY IT MATTERS: The charter's unified cross-format comment model makes anchors polymorphic (pdf-glyph-rect vs md-fuzzy-snippet/CSS-selector); the agent pane is one of the main consumers of comments, and as built it can only describe PDF-style locations to Claude. MD/DOCX/HTML comments fed through this bundle would carry meaningless page fields.
- RECOMMENDATION: When the unified model lands, version the bundle to carry the anchor union (or a pre-rendered location string per anchor kind) instead of bare page numbers, and derive the doc-switch phrasing from the viewer's declared capability rather than extension sniffing. M — but cheap if done together with the shared priming module above.

### session-store breaks the house atomic-write discipline and carries a stale port comment   [severity: low] [effort: S]
- FACT (cited): `saveSessionId` writes with a direct `writeFileSync(pathFor(), …)` (session-store.ts:43) — no temp+rename, unlike the repo-wide discipline the ground-truth map records as established hygiene ("Atomic-write discipline everywhere (temp+rename)", docs/research/2026-06-12-ground-truth-map.md:107). The module header still documents the file as living under "agent-viewer" userData (session-store.ts:4-6), the app this code was ported from (agent-pane-ipc.ts:3 "ported from agent-viewer").
- WHY IT MATTERS: A crash mid-write corrupts session.json; the failure is benign (resume lost, load already catches parse errors at session-store.ts:31-34) but it's an avoidable inconsistency, and the stale path comment will misdirect debugging.
- RECOMMENDATION: Temp+rename write (matching main-process drafts persistence style) and fix the comment. S.

### Stale "adapter is stateless" header contradicts the AdapterState it now requires   [severity: low] [effort: S]
- FACT (cited): adapter.ts:3-4 says "The adapter is stateless; turn/sequence tracking happens in the renderer store", but the same file defines per-session mutable `AdapterState` threaded through every call for streaming-partial correlation (adapter.ts:43-51, 53-55) and claude-backend instantiates it per session (claude-backend.ts:150).
- WHY IT MATTERS: The header is the contract statement future readers trust; it currently asserts the opposite of the design (caller-owned state), which matters because misuse (sharing one state across sessions) would cross-wire streaming message ids.
- RECOMMENDATION: One-line doc fix stating "pure but caller-threads per-session AdapterState". S.
