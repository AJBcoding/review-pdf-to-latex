### IPC result types are declared twice and enforced nowhere — the "typed IPC surface" is vacuous on the result side   [severity: high] [effort: S–M]
- FACT (cited): `EngineResult`, `ResolutionAttempt`/`ResolutionStep`, `PdfHealthReport`, and `PdfHealthResult` are each defined twice, verbatim-parallel: desktop/main/engine.ts:21-68 + 237-253 vs desktop/shared/types.ts:4-80. engine.ts:234-235 even claims "Re-exported from shared/types so callers in main can import from one place" — but the lines below it (engine.ts:237-249) are a local re-definition, and engine.ts imports nothing from @shared (imports are node builtins only, engine.ts:15-19). Nothing links the two copies: the handlers return engine.ts's types through untyped `ipcMain.handle` callbacks (`ipcMain.handle('engine:version', async () => engineVersion())` main/index.ts:360; same shape :366), and on the renderer side `ipcRenderer.invoke(...)` returns `Promise<any>` so the `const electronAPI: ElectronAPI` annotation (preload/index.ts:24) type-checks parameters but vacuously accepts ANY result shape for all ~30 invoke wrappers (e.g. preload/index.ts:25-27).
- WHY IT MATTERS: A change to engine.ts's `EngineResult` (or any handler's return shape in main) compiles cleanly while the renderer keeps consuming the stale shared/types.ts copy — the exact drift class strict-mode TS was adopted to prevent. The duplication is unforced: main/index.ts already imports from '@shared/types' (main/index.ts:18-41), so the alias works in the main project.
- RECOMMENDATION: Delete the engine.ts copies and import from @shared/types (S, mechanical). Then add the missing enforcement once for the whole surface: a `handle<K extends keyof ElectronAPI>(channel, fn)` wrapper (or a channel→{req,res} type map) so `ipcMain.handle` bodies are checked against the same ElectronAPI contract the preload implements (M). This must land before the multi-format work adds docx/html comment channels.

### No runtime validation on any of ~55 channels, and the generic fs read/write channels make the sandbox boundary advisory   [severity: med] [effort: M]
- FACT (cited): The preload declares itself the security boundary ("Anything the renderer can call must be declared here — this is the security boundary", preload/index.ts:22-23) and the window hardening is genuinely strong (`contextIsolation: true, nodeIntegration: false, sandbox: true`, main/index.ts:198-203; CJS-preload-under-sandbox handled in electron.vite.config.ts preload block). But no handler validates its arguments at runtime (e.g. `fs:writeFileText` writes caller-supplied `content` to any caller-supplied absolute path, main/index.ts:588-613; `fs:readPdfBytes` returns the bytes of ANY path, 387-414), and `grep -rn 'will-navigate\|setWindowOpenHandler\|will-attach-webview' main/ preload/` returns nothing — no navigation/window-open guard exists. Channel-name drift, the other classic failure, is currently ZERO: a full channel-set diff between preload and main (`grep -ohE "'[a-z]+:[a-zA-Z]+'" main/index.ts main/claude-pty.ts main/agent-pane-ipc.ts main/results-watcher.ts main/submit.ts main/bundle.ts | tr -d "'" | sort -u` vs same over preload/index.ts, then `comm`) showed no orphans either way (the four `node:*` residuals in the main set are import specifiers matched by the regex, not channels) — but nothing enforces that beyond re-running the grep.
- WHY IT MATTERS: JUDGEMENT: with sandboxed renderers displaying untrusted local HTML/DOCX-derived content (html-viewer/docx-viewer iframes), a renderer compromise escalates directly to arbitrary file read/write via these channels; the missing nav guard is the cheap half of Electron's standard checklist. For a local single-user tool this is hardening, not an incident — hence med, not high.
- RECOMMENDATION: Add `will-navigate` deny + `setWindowOpenHandler(() => ({action: 'deny'}))` in createWindow (S). For validation, don't hand-write zod everywhere: scope fs:writeFileText/readPdfBytes to the opened root + `.review-state/` paths, and fold per-channel runtime checks into the same typed `handle()` wrapper recommended in the first finding (M).

### Quit lifecycle: two independent `before-quit` handlers — subsystem teardown runs during the *prevented* quit and re-runs unawaited on the real one   [severity: med] [effort: S]
- FACT (cited): main/index.ts registers two `before-quit` listeners. The first (282-292) sets `appQuitFlushInProgress`, calls `event.preventDefault()`, awaits the drafts flush handshake for all windows (2 s cap, FLUSH_TIMEOUT_MS 235), then calls `app.quit()` again. The second (818-822) runs `stopResultsWatch(); shutdownClaudePty(); await shutdownAgentPane();`. Electron runs both listeners on the SAME emission, so on the first (prevented) quit the pty/agent subsystems are already torn down while the app keeps running for up to 2 s; on the re-issued quit the second handler runs AGAIN (double shutdown), and its `await` is on an event listener Electron does not await, so `shutdownAgentPane()` races process exit.
- WHY IT MATTERS: Correctness today is coincidental (teardown is presumably idempotent — 3B's to confirm — and the prevented-quit window happens to give the async shutdown time). Any reordering of the two registrations, or a flush that matters to the agent side, breaks silently.
- RECOMMENDATION: Collapse to ONE before-quit handler that preventDefaults, then sequentially `await flushAll(); await shutdownAgentPane(); shutdownClaudePty(); stopResultsWatch();` and only then re-quits via a `quitting` flag. S.

### resolveEngine re-walks the PATH + filesystem probe chain on every engine call, anchored to a nondeterministic cwd   [severity: med] [effort: S]
- FACT (cited): `runEngine` calls `resolveEngine()` unconditionally per invocation (engine.ts:152-153); the chain probes every PATH dir with `access()` (engine.ts:82-90) plus up to 4 cwd-ascending `.venv` candidates (121-128) and the home venv (130-134) — no caching anywhere in the file. Step 3 resolves relative to `process.cwd()` with the in-code admission "In a packaged app, cwd is wherever the user launched from" (engine.ts:119-120). Both IPC consumers re-trigger it: `engine:version` and `engine:pdfHealth` (main/index.ts:360, 366), the latter invoked at every PDF load (renderer/index.ts:1286 via grep).
- WHY IT MATTERS: Per-operation cost is modest today (two channels) but this is the adapter the multi-format direction will multiply (every engine-backed comment operation pays a full re-resolution), and cwd-relative resolution means the packaged app finds different engines depending on launch origin — a support-burden class of bug.
- RECOMMENDATION: Memoize the resolution per app run (invalidate on REVIEW_PDF_ENGINE_PATH change), and resolve the repo-venv step against the doc/project dir handed in by the caller rather than process.cwd(). S.

### API residue and pdf-naming gaps will snag the multi-format direction   [severity: med] [effort: S]
- FACT (cited): (a) `drafts:read`/`drafts:write` still take a dead `_sha256` param (main/index.ts:420, :479) that preload still requires from every caller (preload/index.ts:30-33), while the doc comments in types.ts still describe the pre-migration layout "persisted to `…/drafts/<sha256>.json`" (types.ts:102-103, repeated at :635) — the actual key is path/basename (draftsPathFor, main/index.ts:47-50, whose own comment names .md as the motivating format, :43-46). (b) The external-open argv path accepts `open <path>` for any file but the positional/drag form matches ONLY `.endsWith('.pdf')` (main/index.ts:182), though `classifyFile` knows md/html/docx (68-75). - WHY IT MATTERS: Each is trivial alone; together they mean the backbone's nominal surface says "PDF tool" while its behavior is already multi-format — the next format lands by copy-pasting around the names, and the dead sha256 plumbing keeps a false invariant alive in every signature.
- RECOMMENDATION: One S-sized sweep: drop the `_sha256` params (3-layer change, typecheck-guided), fix the two stale doc comments, rename channel+types to `fs:readFileBytes`/`ReadFileBytesResult` (keep an alias during transition), extend argv positional matching and indexPdfs filtering to the FileKind set (palette ranking can still prefer pdf).
