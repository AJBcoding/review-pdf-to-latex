## Part 1 — Reusable-asset inventory (FACTs cited; per-asset verdict is JUDGEMENT)

### A1. Engine extraction (in-bound PDF adapter)

- FACT: Entry points are `read_annotations` (extract.py:250), `fuzzy_map` (extract.py:538),
  `run_extract` (extract.py:834). The pdfannots subtype IS read — `subtype_name =
  getattr(getattr(raw, "subtype", None), "name", "")` (extract.py:321), kept in a
  function-local `subtypes_by_id: dict[str, str]` (extract.py:308, populated :354) — and
  used for sticky-note→highlight association (`_associate_sticky_notes`, extract.py:176-247,
  which checks `subtypes_by_id.get(ann.id) == "Highlight"` at :215 and `!= "Text"` at :219).
  Its own docstring records the gap: "read loop because :class:`Annotation` does not carry
  subtype" (extract.py:211).
- FACT: The persisted `Annotation` dataclass has exactly eight fields — `id, page, bbox,
  highlighted_text, author, comment, created, trigger_match` (state.py:192-199). No
  subtype, no native PDF annotation id, no reply chain.
- VERDICT (JUDGEMENT): Reusable as the engine-side PDF reader — it already parses subtype
  and stickies and has the bbox-fallback hardening (domain trap #8). Not reusable as-is
  for an Acrobat round-trip: the persisted record cannot distinguish Highlight from
  StrikeOut. Confirms Pass 2's reusability note; schema-v2 of annotations.json is the
  unavoidable cost (persist `subtype`, native id, `in_reply_to`).

### A2. Engine per-comment state machine (workflow substrate)

- FACT: `Status` is an 11-value Literal — pending, applied, accepted, rejected, redrafted,
  deferred, surfaced_pending, surfaced_resolved, needs_review (+ phase/order/method types)
  (state.py:172-183). The legal-transition table is keyed `(status, action)` with actions
  apply/approve/reject/redraft… (state.py:492-510 opened this session; full table runs on).
  `AnnotationState` carries the rich per-comment workflow record: `status, before_text,
  proposed_text, applied_text, applied_at, last_build_id, surface_chat_log, failure_log_path,
  failure_edit_text` (state.py:309-320). Mutator entry points are id-addressed and
  format-free: `set_annotation_status` (apply.py:550), `bulk_surface_pending` (:604),
  `set_current_annotation` (:675), `append_chat_turn` (:706), `record_proposal` (:744),
  `override_mapping` (:773).
- VERDICT (JUDGEMENT): The transition table and the id-addressed mutator surface are the
  most reusable engine assets — nothing in them mentions PDF or LaTeX. The unified model
  can adopt this as the canonical status machine, subject to the vocabulary collision in
  finding F1 below and the schema-guard bypass (Pass 2 high finding) being fixed first.

### A3. main/bundle.ts writer (out-bound PDF adapter — THE adapter template)

- FACT: `writeBundle` (bundle.ts:157) documents its own 6-step pipeline: read source →
  layer Highlights grouped per page with freshly-minted `annot-<n>` ids → serialize +
  sha256 + atomic write → build JSON sidecar with `pdf_annotation_id` stamped back per
  comment → atomic-write sidecar → return id map to the renderer (step comment
  bundle.ts:141-156). The PDF↔JSON link is bidirectional by design: /NM is set to the
  `pdf_annotation_id` "so the JSON sidecar's per-comment link is resolvable from inside
  the PDF too (round-trip safety)" (bundle.ts:56-58). `buildHighlight` (bundle.ts:59)
  builds /Rect + Acrobat-ordered /QuadPoints from `comment.anchor.region` (bundle.ts:67);
  out-of-range pages are skipped softly, not fatally (warn + continue, bundle.ts:210-213).
  IPC delegation is one-line and stateless (`ipcMain.handle('bundle:write', …)` →
  `writeBundleImpl(request)`, main/index.ts:777-781; "Stateless: every write is a fresh
  render" comment :772-776).
- FACT (limits, as shipped): single-region quads only — "Single-region quad in v1 …
  multi-line highlights would push 8 numbers per line" (bundle.ts:73-76); reads
  `comment.anchor.region` / `c.anchor.page` with no anchor-kind narrowing (bundle.ts:67,
  ~:203 — the anchor-blindness is Pass 3's finding, re-confirmed at these lines this
  session).
- VERDICT (JUDGEMENT): This is the strongest single asset in the repo for the capability
  direction: read-native → mint ids → write-native → stamp ids into sidecar → return map
  is exactly the shape of a DOCX comments.xml writer (jszip/OOXML: read docx → insert
  `w:comment` + range markers with minted `w:id` → write → stamp `native_comment_id`).
  Reuse the pipeline shape and the id-stamping contract; gate it behind an anchor-kind
  check (Pass 3 rec).

### A4. shared/bundle.ts (cross-process grammar — already half multi-format)

- FACT: Single source of truth used by both processes by design ("One source of truth so
  PDF annotation /C and the right-drawer card CSS can't drift", shared/bundle.ts:1-6).
  `SOURCE_NAME_RE = /^(.+?)-(\d+)\.(\d+)\.(pdf|md|tex)$/` already parses md and tex
  source versions (shared/bundle.ts:20). But `buildBundleFilename` hard-commits the
  artifact type: "bundles are always PDF + JSON pairs, even when the source is .md or
  .tex; v1 only ships PDF review so the .md/.tex case is moot here" (shared/bundle.ts:34-37).
  `mintBundleId` is the UTC `YYYYMMDD-HHmmss` minting fn (shared/bundle.ts:63-69; its
  duplicate `mintSubmitId` at main/submit.ts:39 is Pass 3's tail finding).
- VERDICT (JUDGEMENT): Filename grammar, palette, and id minting carry over unchanged;
  the "always PDF+JSON" assumption is the one line of this module the unified model must
  renegotiate (see finding F2).

### A5. Anchor polymorphism (the model to fix, plus two reusable legs)

- FACT: Current shape — `AnchorRegion` = page + rect (shared/types.ts:121-124);
  `CommentPayload.anchor: AnchorRegion` REQUIRED on every comment (types.ts:130);
  `md_anchor?` optional five-field blob (types.ts:159-166); discriminator is file-level
  `DraftsFile.anchor_kind?: AnchorKind` (types.ts:190) with exactly two declared kinds
  (`'pdf-glyph-rect' | 'md-fuzzy-snippet'`, types.ts:170). Relocation precedent exists
  twice and is PDF-typed both times: `CommentPayload.new_anchor?: AnchorRegion | null`
  (types.ts:150) and `ResultEntry.new_anchor?: AnchorRegion | null` (types.ts:245).
- FACT: The text-anchor leg is a clean pure module: `MdAnchor` (char_start/char_end/
  prefix/suffix/quoted_text, shared/md/anchors.ts:1-7), `createMdAnchor` with
  CONTEXT_CHARS=40 (:9-26), `fuzzyMatchAnchor(doc, anchor) -> {from, to, confidence:
  'exact'|'fuzzy'|'orphaned'}` (:28-34, :90) — no DOM, no Electron, string-in/string-out.
- VERDICT (JUDGEMENT): Two anchor legs are real and reusable — `AnchorRegion` (PDF) and
  `MdAnchor` + `fuzzyMatchAnchor` (any linear text, incl. DOCX body text and HTML
  extracted text) — once Pass 4B's verification guard lands. Everything AROUND them
  (required PDF anchor, file-level kind, `md_anchor` smuggling, the four-shapes-behind-
  two-kinds drift) is the part the unified model replaces. New fact for the spec: the
  RIG contract (`ResultEntry.new_anchor`) is also AnchorRegion-typed, so the union
  change is a three-party schema event (drafts sidecar + submit/results files + rig),
  not a desktop-internal one — finding F3.

### A6. Drafts sidecar (app-side store)

- FACT: `DraftsFile` = schema_version 1 + doc_version + comments[] + optional anchor_kind
  + optional doc_fingerprint (types.ts:186-192); snapshot semantics, full rewrite per save
  (header comment types.ts:182-185). Path derivation: `<docdir>/.review-state/drafts/
  <basename>.json` (draftsPathFor, main/index.ts:47-50). Rename recovery via
  `DocFingerprint` (title_from_frontmatter / first_500_chars_sha256 / anchor_count /
  last_known_path, types.ts:175-180). Write is atomic tmp+rename (`drafts:write` handler,
  main/index.ts:474-490).
- VERDICT (JUDGEMENT): The sidecar is the right substrate for draft-stage comments in all
  four formats — it is path-keyed JSON that never touches the source file. Known debts
  ride elsewhere (file-level anchor_kind here; draftsCache staleness Pass 4A; fingerprint
  fields computed-not-consulted Pass 3C). Note it shares the engine's `.review-state/`
  directory but NOT its files — see F1 for the absent bridge.

### A7. Results pipeline (round-trip transport — already format-agnostic)

- FACT: `SubmitFile` carries submit_id/doc_id/doc_version(sha256)/source_file_version/
  comments[] with bundle paths OPTIONAL (`bundle_pdf?`, `bundle_json?`, types.ts:211-225).
  `ResultEntry` is id + 5-value `ResultEntryStatus` + reason/new_anchor/agent_note
  (types.ts:230-251); `ResultsFile` adds round_status/new_source_path/version_chosen
  (types.ts:257-270). The watcher pairs purely on strings: `matchesDoc = submit !== null
  && submit.doc_version === state.sha256` (results-watcher.ts:123-124) and replays rounds
  in timestamp order per §10.3 (sort comment + `.sort()`, results-watcher.ts:149-156).
  Submit ops are path/id-only: `promoteDraft` (main/submit.ts:85, writes
  `submit-<id>.json` :89-92,:134), `slingViaGtMail` (:183), `abandonRound` (:310); IPC
  delegations at main/index.ts:783-799.
- VERDICT (JUDGEMENT): Confirms Pass 3C: nothing in the transport knows what a PDF is —
  a DOCX or MD round needs zero changes to promote/sling/watch/match. The two
  format-coupled leaks are `bundle_pdf?/bundle_json?` naming (already optional, so
  generalizing is additive) and `ResultEntry.new_anchor`'s AnchorRegion type (F3).

## Part 2 — Findings (new, this lane)

### F1. The two comment worlds share a directory but not one line of vocabulary — three status enums, zero mapping code   [severity: high] [effort: M — spec work, the code change rides schema-v2]

- FACT (cited): Engine `Status` has 11 values incl. `surfaced_pending`, `surfaced_resolved`,
  `needs_review`, `redrafted` (state.py:172-183). Desktop `CommentStatus` has 7 values
  incl. `needs-followup` and `build_failed` (shared/types.ts:110-117). The rig's
  `ResultEntryStatus` is a 5-value subset of CommentStatus (types.ts:230-236, with the
  explicit note "narrower than CommentStatus"). Verbatim cross-check this session:
  `grep -rn "surfaced_pending\|surfaced_resolved\|needs_review\|redrafted" desktop
  --include='*.ts' --include='*.tsx' | grep -v node_modules` → NO output. Both worlds
  write under the same `.review-state/` dir (draftsPathFor main/index.ts:49;
  submit-<id>.json main/submit.ts:92; engine StateDir per state.py — ground-truth
  architecture fact), and near-miss names exist (`needs_review` vs `needs-followup`,
  underscore vs hyphen; `deferred` in engine Status AND ResultEntryStatus with separately
  defined semantics).
- WHY IT MATTERS: The charter's centerpiece is bridging exactly these two models. Today
  the bridge is not merely missing — the vocabularies have collision-prone near-synonyms,
  so any naive merge silently conflates engine `needs_review` (apply-failure flow) with
  rig `needs-followup` (disposition). The unified spec must publish ONE status set with
  an explicit two-way mapping table before any adapter code is written.
- RECOMMENDATION: Phase B spec must contain a normative status-mapping table (draft in
  Part 3 below) and pick one lexical convention; engine schema-v2 and desktop
  DraftsFile/SubmitFile bumps adopt it together. M.

### F2. The bundle grammar already parses md/tex sources but hard-commits every round to a PDF+JSON artifact pair   [severity: med] [effort: S (spec decision) + M (implementation)]

- FACT (cited): `SOURCE_NAME_RE` accepts `(pdf|md|tex)` extensions (shared/bundle.ts:20),
  and `buildBundleFilename`'s contract comment states "bundles are always PDF + JSON
  pairs, even when the source is .md or .tex; v1 only ships PDF review so the .md/.tex
  case is moot here, but the grammar is the same" (shared/bundle.ts:32-37). Meanwhile
  `SubmitFile.bundle_pdf?`/`bundle_json?` are already optional (types.ts:222-223), and
  the watcher's pairing logic never reads them (match is doc_version-only,
  results-watcher.ts:123-124).
- WHY IT MATTERS: For MD rounds (sidecar-only) and DOCX rounds (the native artifact is a
  .docx with comments.xml, not a PDF), the "bundle" concept must generalize to
  "per-format native artifact + JSON sidecar, both optional". The transport already
  tolerates this (optional fields, pairing ignores them) — only the grammar comment and
  the renderer/main writers assume PDF.
- RECOMMENDATION: In the Phase-B spec, define `bundle` as `{ native_artifact_path?: string,
  sidecar_json_path?: string, format: FileKind }` per round; keep the §10.4 filename
  grammar (it already versions md/tex names). S to spec; the writer generalization is the
  adapter work itself.

### F3. The rig results contract is PDF-typed too — `ResultEntry.new_anchor: AnchorRegion` makes the anchor union a three-party schema event   [severity: med] [effort: S to spec, rides the union migration]

- FACT (cited): `ResultEntry.new_anchor?: AnchorRegion | null` (shared/types.ts:245) and
  `CommentPayload.new_anchor?: AnchorRegion | null` (types.ts:150) — both relocation
  fields are hard-typed to the PDF page+rect shape, written by the RIG ("Set by the rig
  when an `applied` redraft moved the underlying text", types.ts:147-149) and consumed by
  the renderer's reveal path (`c.new_anchor ?? c.anchor`, Pass 4C finding).
- WHY IT MATTERS: Every Pass-3/4 discussion of the discriminated anchor union framed it
  as a drafts-sidecar + renderer change. This line shows the union also changes what the
  RIG must emit for relocated md/docx comments — i.e., the unified-model schema bump must
  be coordinated with the rig-side writer of results files, or relocations stay PDF-only.
- RECOMMENDATION: The Phase-B spec must version SubmitFile/ResultsFile (both already have
  optional `schema_version?: 1`, types.ts:212, 258) and type `new_anchor` as the same
  anchor union, with the rig contract documented as part of the same change. S (spec);
  coordination cost is the real item.

### F4. The complete native-annotation record exists in-repo only as spike output — neither production extraction path persists it   [severity: med] [effort: S (schema fields) once schema-v2 lands]

- FACT (cited): The spike's readback records carry `{page, subtype, rect, contents,
  author, color, quads[]}` (desktop/spikes/rev-cvr-pdf-lib/readback.json:1-42) — i.e.,
  everything needed to repopulate a comment list except engagement level and redraft
  (README.md:141-148). The engine's persisted `Annotation` drops subtype entirely
  (state.py:192-199, gap acknowledged at extract.py:211); the desktop's `CommentPayload`
  has the inverse gap — it carries engagement/redraft/status but its only native link is
  the PDF-only `pdf_annotation_id?` (types.ts:155-156).
- WHY IT MATTERS: The round-trip requirement (display Acrobat's annotations, edit, write
  back, incl. sticky notes and replies) needs ONE record that joins the native fields
  (subtype, native id, author, color, quads, reply parent) with the app fields
  (engagement, redraft, status). Today that record exists only as throwaway spike JSON.
- RECOMMENDATION: Adopt the readback.json field set as the normative `native` block of
  the unified comment record (draft below), with `pdf_annotation_id` generalized to a
  per-format `native_comment_id`. S as schema work; reading/writing it is the adapter
  work owned by 5A's options.

Out-of-scope notes (one line each, for the synthesizer): the spike README's identity
coord transform claim (README.md:26-28) is load-bearing for 5A's pdf.js-annotation-layer
option — cite it there; `mintSubmitId`/`mintBundleId` duplication already filed by Pass 3C.
