## (i) PDF annotation round-trip in the desktop viewer

### Baseline FACTS the options stand on

- FACT (cited): The desktop PDF viewer renders canvas + text layer + struct tree only —
  imports are `TextLayerBuilder, StructTreeLayerBuilder` (desktop/renderer/pdf-viewer.ts:32-33),
  `renderPage` constructs `new TextLayerBuilder({...})` (pdf-viewer.ts:204) — and has ZERO
  annotation-reading surface: verbatim
  `grep -n 'getAnnotations\|AnnotationLayer\|annotationStorage' desktop/renderer/pdf-viewer.ts`
  → no output, exit 1. So annotations already in a PDF are invisible in-app today
  (confirms the ground-truth matrix row "desktop viewer does NOT display PDF annots").
- FACT (cited): The WRITE half is already production code: `buildHighlight` builds the
  full /Highlight dict (bundle.ts:59-108) including round-trip identity `NM:
  PDFString.of(annotationId)` (bundle.ts:104), `appendAnnotsToPage` appends to an existing
  /Annots array (bundle.ts:112-125), and `writeBundle` stamps the same id back into the
  sidecar as `pdf_annotation_id` (shared/types.ts:156; writeBundle pipeline comment
  bundle.ts:141-156). Known v1 fidelity limit: one quad per comment from a bbox —
  "Single-region quad in v1 (renderer captures a bbox, not per-line rects)"
  (bundle.ts:73-76).
- FACT (cited): The READ half is spike-proven but unwired: `testReadback`
  (spike.mjs:278-310) walks `page.node.Annots()` for Highlight/StrikeOut/Underline/
  Squiggly/Text and its real output (readback.json:1-40, opened this session) carries
  page/subtype/rect/contents/author/color/quads — and the spike README states "Existing
  /Highlight annotations can be read back from a PDF by walking `page.node.Annots()`"
  (2026-05-21 spike README, "Result" section). StrikeOut and Text (sticky) dict shapes are
  also spike-proven (README "Annotation dict shape" tables; spike.mjs:125).
- FACT (cited): The display/navigation primitives the read path needs already exist:
  `revealAnchor(page, region)` projects PDF points → viewport and paints a persistent
  highlight (pdf-viewer.ts:288-317), `drawHighlight` merges per-line rects
  (pdf-viewer.ts:447-463), `convertToPdfPoint` round-trips screen→PDF space
  (pdf-viewer.ts:470), and the spike proved the PDF.js↔pdf-lib coord transform is the
  identity (spike README, "Coord transform — proof it's identity").

### Option PDF-A — pdf.js as the reader (getAnnotations), pdf-lib as the writer

- FACT (cited): pdf.js's official API exposes `PDFPageProxy.getAnnotations(params?)` →
  "A promise that is resolved with an {Array} of the annotation objects"
  (https://mozilla.github.io/pdf.js/api/draft/module-pdfjsLib-PDFPageProxy.html, fetched
  this session), and `PDFDocumentProxy` has `saveDocument()` → "A promise that is resolved
  with a {Uint8Array} containing the full data of the saved document" plus an
  `annotationStorage` documented as "Storage for annotation data in forms"
  (https://mozilla.github.io/pdf.js/api/draft/module-pdfjsLib-PDFDocumentProxy.html,
  fetched this session). pdfjs-dist ^5.7.284 is already a dependency
  (desktop/package.json:72).
- JUDGEMENT (trade-offs): The renderer already holds the loaded `PDFDocumentProxy`
  (pdf-viewer.ts:160), so reading native annotations is one `page.getAnnotations()` call
  per rendered page — zero new dependencies, and pdf.js does the parsing the spike does by
  hand (string decoding, inheritance) inside the same object the viewer trusts for
  rendering. Cost: getAnnotations returns pdf.js's view-model, not the raw dicts, so the
  write side still needs a separate library; and pdf.js's own write path
  (annotationStorage + saveDocument) is built around its form/editor storage — the app's
  comment metadata (engagement_level, redraft, status — shared/types.ts:130-156) has no
  home in it, so adopting pdf.js as the WRITER would create a second source of truth
  parallel to the drafts sidecar. Renderer-side read also means annotation data crosses
  IPC to reach main-side writers — but the SelectionPayload already crosses that boundary
  today (spike README "call sequence").

### Option PDF-B — pdf-lib read-back as the reader (extend bundle.ts both ways)

- FACT (cited): Proven by the spike (spike.mjs:278-310; readback.json), pdf-lib ^1.17.1
  already in dependencies (desktop/package.json:71), and the writer it would extend is the
  same library in the same process (bundle.ts:59-125).
- JUDGEMENT (trade-offs): One library, one process (main), one module for the whole PDF
  adapter — read, create, edit, delete all walk the same /Annots array, and matching an
  app-written annotation for edit/delete is direct (`/NM` == `pdf_annotation_id`,
  bundle.ts:104). Cost: hand-parsing breadth — the spike parses 5 markup subtypes and
  simple fields; foreign PDFs bring UTF-16BE strings, missing /T, /IRT reply chains,
  appearance-stream-only annots — each a hand-written case pdf.js's parser already
  handles. Read freshness also requires a main-process re-load of the file the renderer
  has separately loaded (two parses per open).

### Option PDF-C — reuse engine extraction (pdfannots via subprocess)

- FACT (cited): The engine consumes pdfannots as an in-process Python library
  (`import pdfannots`, extract.py:15; `pdfannots.process_file(fh, ...)`, extract.py:303),
  reachable from the desktop only via the `review-pdf extract` subprocess, which is a
  project BOOTSTRAP, not a query: it requires `--pdf` + project dir, refuses to run if
  `.review-state/` exists without `--force` (cli.py:105-120; EXIT_EXISTING_STATE=3,
  cli.py:812), and renders every page to PNG via pdftoppm as a side effect
  (run_extract → `render_pages(pdf_path, pages_dir)`, extract.py:908; pdftoppm invocation
  extract.py:448). Its persisted schema is lossy for round-trip purposes: `Annotation` is
  id/page/bbox/highlighted_text/author/comment/created/trigger_match ONLY
  (state.py:189-199) — no subtype, no quads, no color, no /NM linkage (subtype exists
  transiently in `subtypes_by_id` during extraction, extract.py:209-219, and is dropped).
- JUDGEMENT: Wrong tool for a viewer read path: per-open subprocess + Python-env
  dependency (the 4-step resolution chain, Pass 3's engine.ts findings), state-mutating
  side effects, and a schema that cannot express what the desktop must write back. Where
  the engine extraction DOES stay valuable: it is the only reader with the
  pdfplumber bbox-crop fallback for broken-quad restored PDFs (domain trap #8,
  extract.py:113-156 region) and the sticky→highlight association heuristic
  (extract.py:176-243) — heuristics worth PORTING into the unified adapter, not invoking.

### Recommendation (i) — split read/write across the two libraries already shipped

- RECOMMENDATION (follows from the FACTS above): READ + DISPLAY via pdf.js
  `page.getAnnotations()` inside the existing `renderPage` (the Pass-4 routing note
  anticipated exactly this slot), normalized immediately into the unified comment model
  and rendered through the EXISTING card stream + `revealAnchor` highlight path — NOT a
  second pdf.js annotation DOM, so native and draft comments share one surface (the
  charter centerpiece). WRITE/EDIT/DELETE via pdf-lib in main, extending bundle.ts into a
  `pdf-comments.ts` adapter: create = existing `buildHighlight` + spike's StrikeOut/Text
  shapes; edit = locate dict by `/NM` (app-written) or by (page, annot index) captured at
  read time (foreign annots may lack /NM — write one on first edit); delete = remove ref
  from /Annots. Effort: read+display S–M; write-back edit/delete M. Two flagged risks
  needing a 1-day spike each, not assumptions: (a) reply chains — /IRT round-trip is
  documented PDF 1.7 §12.5.2 territory the spike did NOT exercise ("replies where
  feasible" per charter; treat as feasible-unproven); (b) upgrading the writer from
  single-bbox quads to per-line quads (bundle.ts:73-76) so app highlights survive Acrobat
  round-trip without the bbox-collapse fidelity loss — the capture side already has
  per-line rects (`screenRects`, pdf-viewer.ts:60; merge logic :447-463).
  Reject PDF-C for the live path; port its two heuristics when foreign-PDF breadth lands.

## (ii) DOCX comments.xml read/write

### Baseline FACTS

- FACT (cited): The docx viewer drops comments at display time: `mammoth.convertToHtml`
  is called with `{ styleMap: ['u => em'] }` only (docx-viewer.ts:44-46) — no
  `comment-reference` mapping — and nothing anywhere reads comments.xml (ground-truth
  matrix "comments.xml never read"; no OOXML/zip code exists under desktop/ outside
  node_modules). mammoth's official README: "By default, comments are ignored. To include
  comments in the generated HTML, add a style mapping for `comment-reference`"
  (https://github.com/mwilliamson/mammoth.js, fetched this session).
- FACT (cited): OOXML comment anatomy spans TWO parts — per the python-docx 1.2.0 user
  docs (https://python-docx.readthedocs.io/en/latest/user/comments.html, fetched this
  session): comment content lives in the separate comments-part `word/comments.xml`, while
  the anchor is range markers in the document body ("The start marker appears before the
  run starting with the first character of the range and the end marker appears
  immediately after the run ending with the last character"), and ranges "must enclose
  _contiguous_ runs". CONSEQUENCE the charter wording glosses: "comments.xml read/write
  with body read-only" is exactly true only for EDITING comment text; CREATING or DELETING
  an anchored comment must also insert/remove range-marker elements in document.xml — body
  TEXT stays untouched, but the part is mutated. Any option below carries this; it should
  be stated in the spec, not discovered in review.

### Option DOCX-A — jszip (or fflate) + hand-written OOXML adapter in main

- FACT (cited): The pipeline shape it needs already exists as a template: writeBundle's
  read source → group → write native annotations with minted ids → save → stamp ids into
  sidecar → return id map (bundle.ts:141-156 step comment) — Pass 3's reusability note
  called this "exactly the shape a DOCX comments.xml writer needs". The anchor substrate
  the unified model needs (quoted_text + prefix/suffix text anchors, not CSS selectors)
  was already recommended by Pass 4B against this lane's requirement (REVIEW.md,
  "html-viewer and docx-viewer are ~60% verbatim twins…": for the DOCX round-trip the
  selector anchors "are the wrong substrate — `nth-of-type` paths against
  mammoth-regenerated DOM, with no stable mapping back to OOXML ranges").
- JUDGEMENT (trade-offs): Pros — exact scope (read comments + ranges; write comment +
  markers; edit text; delete), no new runtime, lives in main beside the PDF adapter,
  text-anchor resolution is a linear scan over document.xml run text (same
  TextQuoteSelector logic as md). Cons — hand XML: namespace bookkeeping, comments-part +
  content-types + .rels registration when the doc has no comments yet, run-splitting when
  an anchor starts mid-run (the contiguous-runs rule above), and replies/resolution live
  in a third part (commentsExtended) that is where DIY cost explodes — scope v1 to flat
  comments, exactly the line python-docx itself drew (next option). One new small dep
  (jszip or fflate; mammoth uses a zip internally but exposes no zip API — package.json:69
  lists only mammoth).

### Option DOCX-B — existing libraries

- FACT (cited): JS `docx` (dolanmiu) is a GENERATOR — README: "Easily generate and modify
  .docx files with JS/TS with a nice declarative API" (https://github.com/dolanmiu/docx,
  fetched this session); no documented parse/read of an existing file's comments — it
  cannot do the READ half. mammoth can only surface comments as rendered HTML appended "to
  the end of the document" via the `comment-reference` style map (mammoth README, fetched)
  — a display affordance, not a model API, and no write. Python `python-docx` 1.2.0 has a
  REAL comments API — `document.comments`, `document.add_comment(runs=..., text=...,
  author=..., initials=...)`, comment id/author/date/text properties — but with two
  documented limits: no threading/resolution ("Neither of these features is supported by
  the initial implementation") and, per the release notes, no access to the comment
  reference range — the anchored text (python-docx user docs + PyPI/release summary,
  fetched this session).
- JUDGEMENT: No JS library on the table does read+write of comments on an existing file;
  python-docx is the one credible library route, but it (a) drags the Python engine env
  into the only format path that is currently pure-desktop (and the desktop must work when
  engine resolution fails — Pass 3 engine.ts findings), (b) cannot read the anchored
  range, which the card stream needs for quoted_text and reveal, so document.xml parsing
  by hand happens ANYWAY, and (c) puts the DOCX adapter in a different process and
  language from the PDF adapter, doubling the adapter-contract surface. As a WRITE-only
  fallback it remains the cheapest proven `add_comment` if hand-written marker insertion
  stalls — a hedge, not the plan.

### Option DOCX-C — re-litigating the SuperDoc rejection under the narrower requirement

- FACT (cited): The 2026-05-16 rejection was decided on FILE-FORMAT grounds: "Our problem
  deals with two file types neither of which is DOCX" and disposition "Don't revisit
  unless the source-of-truth format changes from LaTeX to DOCX"
  (docs/research/2026-05-16-superdoc-fit-analysis.md, "Why it doesn't fit" +
  "Disposition"). That ground no longer holds: DOCX is now a first-class reviewed format
  (charter objective 2). Current SuperDoc capability, per official docs
  (https://docs.superdoc.dev/modules/comments, fetched this session): comments module with
  `readOnly`, `allowResolve`, `documentMode: viewing` ("Comments are hidden by default
  when documentMode is viewing" — visibility togglable), full comment API (`addComment`,
  `addCommentReply`, `resolveComment`, `removeComment`, `onCommentsUpdate` events), and
  DOCX export round-trip (`commentsType: "external"` preserves comments). Interop caveat
  on record: SuperDoc failed to display python-docx-authored comments until 2026
  (github.com/superdoc-dev/superdoc issue #752, closed via PR #1505, fetched this
  session). License: AGPLv3 + commercial (superdoc-fit-analysis.md), acceptable for
  personal scope per that doc.
- JUDGEMENT: On pure DOCX-comment capability SuperDoc now EXCEEDS every other option —
  replies and resolution included, which DOCX-A/B both punt. But it fails the
  architecture test the charter sets: it is a ProseMirror+Yjs EDITOR with its own comment
  store and its own sidebar UI, while the centerpiece requirement is ONE unified model
  rendered through the app's existing card stream (Pass 4C: the card stream is the single
  surface; Pass 3: drafts sidecar is the store). Embedding it means: replace the mammoth
  iframe with a full editor locked to viewing mode, run a parallel comment store
  synchronized to the drafts sidecar via `onCommentsUpdate` + export, accept its
  DOM/theming, and inherit its OOXML interop history (issue #752 class) — for one of four
  formats. Verdict: KEEP the rejection, but re-ground it — the 2026-05-16 format rationale
  is obsolete; the standing rationale is model-ownership and footprint. Record the
  revisit trigger honestly: if owner requirements grow to threaded replies + resolution +
  tracked changes on DOCX, hand-rolling commentsExtended is the point where DOCX-A's cost
  curve crosses SuperDoc's, and this analysis should be redone.

### Recommendation (ii)

- RECOMMENDATION: DOCX-A — a main-process `docx-comments.ts` adapter on jszip/fflate,
  v1 scoped to flat comments (no replies/resolution, matching python-docx's own scoping
  precedent): READ = parse comments.xml + locate range markers in document.xml, emit
  unified comments with quoted_text from the enclosed runs (M); WRITE = mint w:id, append
  to comments.xml (creating part + rels when absent), insert range markers with
  run-splitting, resolve anchors by TextQuoteSelector scan over run text (M); EDIT/DELETE
  = comments.xml text swap / comment + marker removal (S, rides the same module). Display:
  add mammoth's `comment-reference` style map so native comment positions are visible in
  the iframe (S, independent). Hedge: python-docx write-only via the engine if marker
  insertion proves nasty. Precondition (both formats): the discriminated anchor union —
  next section.

## (iii) Unified cross-format comment-model shape

### The options, against the cited current shape

- FACT (cited, consolidating prior passes + re-opened this session): today's shape is a
  REQUIRED PDF anchor on every comment (`anchor: AnchorRegion`, shared/types.ts:130), an
  optional bolt-on `md_anchor?` (types.ts:159-166), a FILE-level two-value discriminator
  (`AnchorKind`, types.ts:170; on DraftsFile, types.ts:186-192), html/docx anchors
  smuggled through `md_anchor` `as any` (Pass 4B, renderer/index.ts:2214-2223), and a
  writer that consumes the PDF anchor blind (bundle.ts:203-204). Native round-trip
  identity exists for PDF only, as `pdf_annotation_id` (types.ts:156).
- Option M-1 (status quo trajectory — accrete): add `docx_anchor?`, `html_anchor?`, more
  file-level flags. REJECTED on the evidence above — three passes independently traced
  live bugs to exactly this pattern (bundle blind read, "p.1 · 0,0 0×0" cards, as-any
  smuggle); a fourth optional field scales the bug surface, not the capability.
- Option M-2 (per-comment discriminated anchor union + per-format adapters): the shape
  Pass 3 named "the highest-leverage pre-work for the unified model" and Pass 4 evidenced
  end-to-end. This lane's round-trip analysis adds three requirements the union must
  carry that prior passes didn't need: (a) the PDF kind must hold `quads:
  Rect[]` (plural), not one bbox — native annots read back with per-line quads
  (readback.json `quads` arrays; opened this session) and the current single-quad write is
  an acknowledged v1 limit (bundle.ts:73-76); (b) a per-comment `origin`/provenance field
  (`app-draft | native-pdf | native-docx | engine-extract`) so round-trip imports are not
  re-written as duplicate annotations on the next bundle save (writeBundle regenerates
  from scratch every save — "the bundle PDF is regenerated from scratch on every write",
  bundle.ts:152) and so engine annotations.json rows
  (state.py:189-199) map in without schema change; (c) a per-comment `native_ref` for
  edit/delete targeting — PDF: /NM or read-time object index (bundle.ts:104 precedent);
  DOCX: `w:id`. The text-anchor kind is ONE shape shared by md/html/docx
  (quoted_text/prefix/suffix + char positions), per Pass 4B's consolidation
  recommendation.
- Option M-3 (adopt W3C Web Annotation Data Model): the spec's TextQuoteSelector is
  exact/prefix/suffix and TextPositionSelector is start/end
  (https://www.w3.org/TR/annotation-model/, fetched this session) — the existing
  `md_anchor` (char_start/char_end/prefix/suffix/quoted_text, types.ts:159-166) is already
  field-for-field isomorphic to the PAIR of them. Full adoption (JSON-LD, IRIs, bodies vs
  targets) is over-general for a single-user sidecar file. The one pattern worth
  importing: "Multiple Selectors SHOULD select the same content" — selector redundancy is
  exactly the robustness fix Pass 4B demanded for fuzzy re-anchoring (keep the original
  quote immutable, store relocations separately).
- RECOMMENDATION: M-2 with M-3's vocabulary: a `shared/comments.ts` v2 schema —
  per-comment `anchor: PdfQuadAnchor | TextQuoteAnchor | …` discriminated union (kinds
  named truthfully, html-selector demoted to a locality HINT field per Pass 4B), plus
  `origin` and `native_ref`; DraftsFile `schema_version: 2` with a tolerant reader
  migrating v1 rows (the file-level `anchor_kind` becomes derivable and is dropped);
  adapters per format implementing one interface (read native → model; write model →
  native; capabilities record), with bundle.ts as the first adapter retrofitted. Effort: M
  for schema+migration+bundle retrofit (consistent with Pass 3's sizing), each new adapter
  priced separately above. The 5B lane's draft spec is the natural home for the field
  list; this lane's contribution is the three round-trip-driven requirements (quads[],
  origin, native_ref) that the union must not ship without.

### Out-of-scope notes (one line each)

- The charter phrase "DOCX: comments.xml read/write, body read-only" should be amended in
  Phase B to acknowledge range-marker writes into document.xml (fact cited in (ii)).
- pdf.js AnnotationEditorLayer as an editing UX was not deeply evaluated: its storage
  model is form/editor-scoped per official docs ("Storage for annotation data in forms");
  if Phase B wants in-place visual annotation editing rather than card-driven editing,
  that is a separate UX spike — nothing in the charter requires it.
- The engine's sticky→highlight association and bbox-crop fallback heuristics
  (extract.py:176-243, :113-156 region) are port candidates for the PDF adapter's
  foreign-file robustness — noted for 5B's reusable-assets inventory.

## Top 3 in this lane

1. **PDF round-trip: read with pdf.js `getAnnotations` in the existing `renderPage` slot,
   write/edit/delete with pdf-lib in main extending bundle.ts** — both libraries already
   shipped and spike-proven on their respective halves (pdf-viewer.ts:204 grep-confirmed
   annotation-free today; bundle.ts:104 /NM identity already written; spike.mjs:278-310 +
   readback.json prove read-back; official pdf.js API confirms getAnnotations/
   saveDocument). Reject engine-extraction reuse for the live path (subprocess bootstrap +
   lossy schema, cli.py:105-120, state.py:189-199). Spike before committing: /IRT replies,
   per-line quad upgrade.
2. **DOCX: hand-rolled jszip+OOXML adapter in main, v1 flat comments; keep the SuperDoc
   rejection but on NEW grounds** — the 2026-05-16 format rationale is obsolete (DOCX is
   now in scope); the standing rationale is model-ownership + footprint vs the unified
   card stream, with an explicit revisit trigger (threads/resolution requirements). No JS
   library reads+writes existing-file comments (docx = generator; mammoth = display-only);
   python-docx 1.2.0 is the write-only hedge. Load-bearing spec fact: comment CREATE/DELETE
   must touch document.xml range markers — "comments.xml only" is true only for edits.
3. **Unified model: per-comment discriminated anchor union (M-2) with W3C selector
   vocabulary, carrying three round-trip-driven fields prior passes didn't surface —
   `quads[]` (not bbox), `origin` provenance, and `native_ref`** — without which round-trip
   imports duplicate on every bundle regeneration (bundle.ts:148-151) and edit/delete has
   no target handle. This is the precondition for both adapters above and the direct
   continuation of the Pass-3/Pass-4 discriminated-union findings. (Cross-ref: bundle.ts:152
   "regenerated from scratch on every write".)
