### Viewer auto-reload contract (HEAD /api/state) is unimplemented — the legacy viewer's core loop is broken   [severity: high] [effort: S]
- FACT (cited): `templates/frame.html:666-669` documents the reload mechanism: "Polls HEAD /api/state every 500ms (spec §10.2); on Last-Modified change, reloads. Server contract (chunk D): HEAD /api/state must set Last-Modified to the state.json mtime, or ETag to its md5." The poll at frame.html:672-679 computes `tag = r.headers.get("etag") || r.headers.get("last-modified") || ""` and reloads only when the tag changes. But server.py implements only `do_GET` (server.py:265) and `do_POST` (server.py:288); `grep -rn 'do_HEAD' src/ tests/` returns no hits, and `grep -rn 'Last-Modified\|ETag\|etag' src/review_pdf_to_latex/server.py tests/test_server.py` returns nothing. `_send_bytes` (server.py:674-680) sends only Content-Type/Content-Length/Cache-Control. Because `ReviewHandler` subclasses `http.server.SimpleHTTPRequestHandler` (server.py:255), HEAD requests fall through to the stdlib's filesystem handler rooted at the server process cwd, which 404s `/api/state` with no ETag/Last-Modified header.
- WHY IT MATTERS: The tag is permanently `""`, so the page never auto-reloads — the entire click→engine→reload loop the in-page comments describe (frame.html:462-465: "POST to /api/events, then wait for state.json to change (the 1s poll below picks that up and reloads)") cannot fire. Buttons disable themselves on click (frame.html:540-543) and only the no-consumer watchdog re-enables them. test_server.py (1,554 LOC, 68 tests) never issues a HEAD request (grep above), so this rot is invisible to the suite. Secondary: the SimpleHTTPRequestHandler fallthrough means `HEAD /<anything>` probes arbitrary files under the server cwd (existence + size + mtime disclosure; localhost-only, but unintended).
- RECOMMENDATION: Decide keep-vs-kill first (see last finding). If kept: implement `do_HEAD` routing `/api/state` to a stat-based Last-Modified/ETag responder and switch the base class to `BaseHTTPRequestHandler` to close the fallthrough (S). If killed/frozen: record this as the headstone evidence that the stack has already rotted.

### `_render_frame` mixes navigation policy with template context, and duplicates the visibility filter inside the same file   [severity: med] [effort: S]
- FACT (cited): `_render_frame` (server.py:415-534, ~120 LOC) inlines: unresolved-status filtering via an `_is_unresolved` closure (server.py:450-459, with the comment "Unknown status — treat as unresolved so it remains visible"), default-annotation selection `_pick_default_id` (server.py:461-467), saved-id fallback logic (server.py:469-487), and only then Jinja context assembly (server.py:498-534). `_resolve_navigate_target` (server.py:172-231) re-implements the identical visibility rule as its own `_visible` closure (server.py:200-211, comment: "Unknown status — keep visible so it isn't silently skipped") plus its own first-visible fallback (server.py:216-220).
- WHY IT MATTERS: Two copies of "which annotations are visible and which is current" in one module is exactly how the viewer and the navigate endpoint drift apart (e.g., `_pick_default_id`'s surface-first ordering exists only in the render path, not the navigate path — server.py:461-467 vs 216-220). This is the ground-truth map's `_serve_frame` suspicion, located precisely: the mixing is in `_render_frame`, not `_serve_frame` (which is 10 lines, server.py:404-413).
- RECOMMENDATION: Extract a single `resolve_visible(state, annotations, view) -> (visible_ids, current_id)` helper used by both `_render_frame` and `_resolve_navigate_target`; `_render_frame` shrinks to context assembly (S).

### Atomic-write and state-dir constants duplicated; the "module independence" rationale is already false   [severity: med] [effort: S]
- FACT (cited): server.py:1060-1061 defines `_atomic_write_state` with docstring "Local atomic write — mirrors state.atomic_write_json from chunk A"; its body (server.py:1064-1082) is line-for-line the same mkstemp/fsync/os.replace logic as `state.atomic_write_json` (state.py:66, body confirmed via sed this session). server.py:37 defines `STATE_DIR_NAME = ".review-state"  # mirrors state.STATE_DIR_NAME for module independence` — but `grep -n 'STATE_DIR_NAME' src/review_pdf_to_latex/state.py` shows no such constant exists (the canonical spelling is `StateDir.DIR_NAME` at state.py:29), and server.py:34 already imports `status_is_terminal` from state, so the independence claim is void. build.py:319 and preview.py:51,54 hard-code the `".review-state"` literal instead.
- WHY IT MATTERS: Two atomic-write implementations means a durability fix (e.g., directory fsync) lands in one and not the other. The dangling `state.STATE_DIR_NAME` comment misleads the next reader; four spellings of the state-dir constant across the serve/build surface alone is the inconsistent-access-style pattern the hunt list targets.
- RECOMMENDATION: Delete `_atomic_write_state`, import `atomic_write_json`; hoist `STATE_DIR_NAME` to state.py (or use `StateDir.DIR_NAME`) and reference it everywhere (S).

### preview() returns a guessed build ID — read-then-rebuild TOCTOU   [severity: med] [effort: S]
- FACT (cited): `_invoke_build` (preview.py:186-195) reads state.json and computes `build_id = build_mod.next_build_id(state)` BEFORE calling `run_build_command`, which independently re-reads state and recomputes its own ID (build.py:337-338: `state = read_json(state_path)` / `build_id = next_build_id(state)`); nothing flows the actual ID back — `run_build_command` returns only an int exit code (build.py:298). `next_build_id` is `len(state["builds"]) + 1` (build.py:129-138).
- WHY IT MATTERS: Any builds[] append between the two reads makes `preview()` return an ID pointing at the wrong artifact — the viewer then displays the wrong speculative build. The window spans a full LaTeX compile (up to 2 passes × 120 s, build.py:79 `_LATEX_PASSES = 2`, build.py:349 `timeout_sec=120`). preview() also ignores `run_build_command`'s failure exit code entirely, so a failed compile still returns a "successful-looking" build ID.
- RECOMMENDATION: Change `run_build_command` to return (exit_code, build_id) — or add a thin `run_build(...) -> BuildResult` API the CLI wraps — and have preview() use the real ID and propagate failure (S).

### state.json read-modify-write has no inter-process lock (events file does)   [severity: med] [effort: M]
- FACT (cited): The events log is flock-serialized (`_append_event_line`, server.py:239-241: `fcntl.flock(fd, fcntl.LOCK_EX)`), but state.json writers in this lane are bare read-modify-write: build.py:337 `state = read_json(state_path)` … build.py:391-392 `state.setdefault("builds", []).append(entry)` / `atomic_write_json(state_path, state)`; handle_serve persists `--order` the same way (server.py:1111-1114: `json.loads(state_path.read_text())` then `_atomic_write_state`). apply.py also writes state.json (out of scope, noted for 2A).
- WHY IT MATTERS: Atomic rename prevents torn files but not lost updates: `review-pdf build` racing a serve-side or apply-side write silently drops one side's mutation (e.g., a builds[] entry vanishes, desyncing `next_build_id`'s len()-based numbering and orphaning `builds/build-NNN.pdf` artifacts). Low probability for a solo CLI today, but the charter's unified comment model puts the desktop app and engine on this file concurrently.
- RECOMMENDATION: Add a `state.lock` flock around read-modify-write cycles (one helper in state.py, used by build/serve/apply), mirroring the existing events-file discipline (M).

### Speculative in-place edit can destroy concurrent edits and leaves no on-disk snapshot across a crash   [severity: med] [effort: M]
- FACT (cited): `with_in_place_edit` snapshots the .tex file only in memory (preview.py:121 `original_bytes = latex_file.read_bytes()`), writes the speculative text to the real file (preview.py:126), runs the build, then unconditionally restores `original_text` in `finally` (preview.py:131-132). The recovery dump is written only when the restore raises OSError (preview.py:133-140). The mutation window spans the compile (up to 2×120 s, build.py:79,349).
- WHY IT MATTERS: (a) Any external edit to the file during the window — the owner's editor, or a future desktop editing surface — is silently overwritten by the restore. (b) On SIGKILL/power-loss mid-build, `finally` never runs and the file is left containing speculative text with the only snapshot lost with the process; recovery is "from git or backup" (the message at preview.py:146-147 admits this for the dual-failure case, but the crash case has no message at all).
- RECOMMENDATION: Write the snapshot to `.review-state/preview-recovery-…` BEFORE mutating (delete on clean restore), and compare file bytes to the speculative text before restoring, refusing to clobber third-party changes (M).

### `wait-event --since` re-call contract drops same-second events   [severity: med] [effort: S]
- FACT (cited): Event timestamps are second-resolution (`_utc_now_iso`, server.py:87-93: `isoformat(timespec="seconds")`); `wait_for_events` filters strictly `if ts > since_ts` (server.py:932); the documented compaction-recovery pattern is "the post-compaction re-call (with --since <last_observed_ts>) to be able to pick up any concurrent event" (server.py:1020-1022).
- WHY IT MATTERS: Two clicks inside the same wall-clock second straddling a consumer restart: the re-call with `--since` equal to the first event's ts skips the second event (strict `>` on an equal string). The consumer (Claude skill) silently never processes that action — and the viewer's only feedback channel is the broken reload loop above. Minor adjacent contract drift: wait_for_events claims "Side effects: NONE" (server.py:851) but the kqueue watcher `touch`es the events file when absent (server.py:977-980).
- RECOMMENDATION: Emit sub-second timestamps (timespec="microseconds") or switch the cursor to a byte offset / event sequence number; one-line ts change + test (S).

### Keep-vs-kill: split the format-agnostic event bus out of server.py, then freeze or retire the viewer   [severity: high] [effort: M]
- FACT (cited): server.py is two unrelated programs in one module: (1) the HTML viewer (`ReviewHandler` + `_render_frame` + page/build/static serving, server.py:255-690) and (2) a format-agnostic event pipeline — `_validate_event` (server.py:96-169), `_append_event_line` with flock (server.py:234-252), `wait_for_events` + watchers (server.py:838-1004), `handle_wait_event` (server.py:1179-1223) — which is the consumer API the CLI wires (cli.py:325-327 imports `handle_serve`; the wait-event path is the skill integration per server.py docstring :1-13). The viewer half is superseded by the Electron app (ground-truth map "two parallel viewer stacks") and demonstrably rotted (reload + diff2html findings above). Reusable-for-capability inventory verified this session: `pdf_health.py` is a pure `(pdf_path) -> (exit_code, report)` function with its own `schema_version` (pdf_health.py:65,166-167) and the desktop ALREADY shells into it (`desktop/main/engine.ts:265`: `runEngine(['pdf-health', '--pdf', pdfPath, '--json'] …)`) — the one engine module in this lane with a live cross-layer consumer and the best adapter precedent for the unified model. `compute_page_md5s`/`paginate_diff` (build.py:164-238) are PDF-generic and LaTeX-free. terminal.py (456 LOC of hand-rolled RFC 6455 + pty, terminal.py:9-14 justifies the no-dependency choice) exists ONLY to serve the legacy viewer's embedded pane and duplicates desktop/main/claude-pty.ts across layers.
- WHY IT MATTERS: The capability direction needs the engine's event/JSONL pattern (closest existing thing to a cross-layer comment-event bus) and pdf_health's clean adapter shape — not the HTML viewer or its WebSocket terminal. Today the survivable 40% is fused to the rotting 60% in one 1222-LOC file, and terminal.py + 831-line frame.html + vendored xterm are pure carrying cost for a UI the owner no longer drives.
- RECOMMENDATION: Extract `_validate_event`/`_append_event_line`/`wait_for_events`/`handle_wait_event` (+ signal sentinels) into an `events.py` module with zero viewer imports (M); then make the explicit Phase-B call between (a) deleting server-viewer + terminal.py + templates (~2.6K LOC + 283 KB vendor) keeping only `wait-event`, or (b) freezing them behind a "legacy, headless-only" README note with the reload bug fixed. Evidence here favors (a) unless a headless use-case is documented.
