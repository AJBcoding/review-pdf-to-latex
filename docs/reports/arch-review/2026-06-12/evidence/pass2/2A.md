### apply_edit / revert_edit mutate the .tex file BEFORE status validation — reachable partial-write corruption   [severity: high] [effort: S]
- FACT (cited): In `apply_edit`, the .tex file is rewritten at apply.py:300–305 (`tex_path.open("w")` + `writelines`), but `validate_status_transition(current_status, "applied", "apply")` runs only afterwards at apply.py:334–338, raising `IllegalStatusTransitionError` before the state.json/mapping.json writes at apply.py:341–342. The transition table has no `("accepted", "apply")`, `("deferred", "apply")`, or `("surfaced_resolved", "apply")` keys (state.py:496–501 enumerate only pending/applied/rejected/redrafted/needs_review/surfaced_pending sources), so calling `apply_edit` on e.g. an `accepted` annotation raises. `revert_edit` has the same shape: file write at apply.py:487–491, validation at apply.py:512–517, state writes at apply.py:529–530.
- WHY IT MATTERS: On a validation failure the .tex file is already mutated but `before_text` was never persisted and mapping line-shifts were never recorded — the engine has no record the edit happened and cannot revert it. This violates the module's own contract ("All writes go through state.atomic_write_json and all status transitions go through state.validate_status_transition", apply.py:4–7) by ordering the unguarded write first.
- RECOMMENDATION: Hoist the `validate_status_transition` call above the file mutation in both `apply_edit` and `revert_edit` (the inputs — current status — are already in hand at apply.py:250/448). Pure reordering, no behavior change on the happy path. Size S.

### commit_phase stages .gitignore'd `.review-state/` files without `-f` — likely exit-19 on every extract-bootstrapped project   [severity: high] [effort: S]
- FACT (cited): `run_extract` step 8 calls `ensure_gitignore_entry(project_dir, entry=".review-state/")` (extract.py:945; default entry also `.review-state/` at extract.py:743–746), writing `.review-state/` into the project's .gitignore. `commit_phase` then stages `state.json`, `mapping.json`, `annotations.json`, and `state-events.jsonl` from under `.review-state/` (commit.py:307–314) via `git add -- <paths>` with no `-f` (commit.py:316–327), raising `CommitFailedError` (exit 19) on nonzero return. The commit test fixture `_init_project_repo` (tests/test_commit.py:145–215, opened) creates the repo and `.review-state/` by hand and never writes a .gitignore, so the interaction is untested.
- WHY IT MATTERS: JUDGEMENT: `git add` on explicitly-named untracked ignored paths exits nonzero ("Use -f if you really want to add them"), so on any real project bootstrapped by `review-pdf extract`, `commit-phase` should fail with exit 19 — the extract-side gitignore policy and the commit-side staging policy contradict each other, and the test gap hides it. (Flagged for a Pass-6 verifier or a quick repro; the two cited code paths are unambiguous, the git behavior claim is the only uncited link.)
- RECOMMENDATION: Decide which policy wins: either stop staging `.review-state/*` (commit message already points at the snapshot path, commit.py:187) or stage with `git add -f`, and add a regression test whose fixture includes the extract-written .gitignore. Size S.

### Schema-version guard is bypassed on every MUTATING read — migration design can't work   [severity: med] [effort: S]
- FACT (cited): `state.read_json` enforces `schema_version` (missing → SchemaVersionError, newer → SchemaVersionError, older → MigrationRequiredError; state.py:148–163). But apply.py defines its own `_read_json` that is a bare `json.load` (apply.py:121–123) and uses it for every mutator read (`_load_state_and_mapping` apply.py:148; direct uses at apply.py:584, 646–647, 699, 730, 766, 803). commit.py reads state.json and mapping.json with raw `json.load` too (commit.py:281–282, 226–227). `assert_source_pdf_unchanged` also raw-loads annotations.json (state.py:643–644). Verbatim survey: `grep -rn 'read_json\|json.load' src/review_pdf_to_latex/*.py` shows `state.read_json` used only by build.py:337, preview.py:186/229, status.py:92 — i.e., only read-only/secondary paths. migrate.py is an empty registry stub whose design depends on the read-time guard tripping (`migrate.py:13–15`: a partial migration "still trips the 'needs migration' check on next read").
- WHY IT MATTERS: The modules that WRITE state (apply, commit) are exactly the ones that never check `schema_version`, so a future v2 state file would be blindly mutated and rewritten by a v1 engine — the entire `migrate-state` safety story (SUPPORTED_SCHEMA, MigrationRequiredError, exit 14) only protects the harmless read-only paths. This also matters for the unified comment model, which is the named driver of the next schema bump (ground-truth map hotspot 16).
- RECOMMENDATION: Delete `apply._read_json` and route apply.py/commit.py reads through `state.read_json` (wrap SchemaVersionError/MigrationRequiredError into ApplyError/CommitError subclasses for exit-code mapping, matching the existing `_guard_source_pdf` wrapping pattern at apply.py:79–93). Size S.

### Revert after an empty-text apply (deletion) destroys one unrelated line — documented special case nobody handles, untested   [severity: med] [effort: S]
- FACT (cited): `apply_edit` with empty `new_text` deletes lines and records a degenerate range with the in-code caveat "callers must treat empty edits as a special case" (apply.py:308–313: `map_entry["line_range"] = [start, start]` when `new_lines` is empty). `revert_edit` does not special-case it: it computes `current_count = end - start + 1` (= 1 for the degenerate range) and writes `all_lines[: start - 1] + before_lines + all_lines[end:]` (apply.py:484–485), which removes the line now at position `start` — a line that was never part of the original edit (after the deletion, the formerly-following line occupies that slot) — and `line_shift = len(before_lines) - 1` (apply.py:493) under-shifts subsequent mappings by one. No test exercises empty text: `grep -rn 'new_text=""\|empty' tests/test_apply.py` returns nothing, and every `new_text=` in tests/test_apply.py is non-empty (test list at tests/test_apply.py:123–605, opened).
- WHY IT MATTERS: A latent file-corruption bug in the revert path of a legal apply input; both the swallowed line and the off-by-one mapping shift would be silent.
- RECOMMENDATION: Either reject empty `new_text` in `apply_edit` (simplest; deletion-by-edit can be expressed as replacing with the surrounding line) or encode an explicit empty-range marker and special-case it in `revert_edit`; add tests for apply-empty → revert. Size S.

### Half-adopted typed layer: state.py dataclasses are write-only, and the raw-dict mutators have drifted from them   [severity: med] [effort: M]
- FACT (cited): The dataclasses (`Annotation`, `Mapping`, `AnnotationState`, `StateFile`, ~240 LOC of hand-written from_dict/to_dict at state.py:189–430) are constructed only by extract.py's bootstrap path — `grep -rn 'StateFile\|AnnotationState\|MappingCandidate\|Mapping\.from_dict\|Mapping(' src/review_pdf_to_latex/ --include='*.py' | grep -v 'state.py:'` matches only extract.py (imports at extract.py:20–23; constructions at extract.py:571–733). All mutators operate on raw dicts (e.g., apply.py:322–339, 589–599). Three concrete drifts: (a) `set_annotation_status` writes a `last_status_reason` key (apply.py:598–599) that `AnnotationState` does not define (field list state.py:313–321; to_dict state.py:343–359 — round-tripping through the dataclass would silently drop it); (b) `override_mapping` sets `entry["candidates"] = None` (apply.py:836) while `Mapping.from_dict` iterates `d.get("candidates", [])` (state.py:275), which raises TypeError when the key exists with value None — and its own docstring says "clears any candidates[]" (apply.py:783–784); (c) `Annotation.created` is typed `str` (state.py:198) but is populated from `_format_created` whose signature is `-> str | None` (extract.py:28, used at extract.py:363).
- WHY IT MATTERS: The schema source of truth is split between dataclasses (write/bootstrap side) and ad-hoc dict keys (mutate side), and they already disagree in three places. The unified cross-format comment model (charter centerpiece) will need a single authoritative schema layer; today there isn't one.
- RECOMMENDATION: Pick a direction before the schema-v2 work: either make the mutators round-trip through the dataclasses (and add the missing `last_status_reason` field, fix `candidates=None`→`[]`) or demote the dataclasses to bootstrap-only builders and document the dict keys as the schema. Fix drifts (a)–(c) immediately regardless. Size M (drift fixes alone: S).

### fuzzy_map re-reads and re-scores the entire .tex tree for EVERY annotation, with per-window re-normalization   [severity: med] [effort: M]
- FACT (cited): `fuzzy_map` takes a single annotation and, inside the call, enumerates and reads every .tex file (`for path in _enumerate_tex_files(...)`: read_text at extract.py:586–588); `run_extract` calls it in a loop over all annotations (extract.py:915–916). Inside, for every window start it re-joins and re-normalizes the growing window from scratch — `_normalize(_strip_latex(" ".join(cum_text)))` runs in the innermost loop for each (start, end) prefix (extract.py:610–611) — and EVERY scored window is appended to a global `all_windows` list (extract.py:584, 615) that is fully sorted at the end (extract.py:630) although only the global best (extract.py:631) and best-per-file top-3 (extract.py:645–656) are consumed.
- WHY IT MATTERS: Cost is O(annotations × files × lines × window-length) with redundant disk reads and normalization; for a long review (dozens of annotations on a multi-file thesis) this is the extract bottleneck. More strategically: an "index the corpus once, then resolve many snippets" decomposition is exactly the shape a unified cross-format anchor resolver needs (the desktop's `md-fuzzy-snippet` anchors solve the same problem), so the refactor pays twice.
- RECOMMENDATION: Split into `build_window_index(root) → per-file normalized windows` (computed once per run_extract, with incremental per-line normalization) + `resolve(target, index) → Mapping`; track best-overall and best-per-file during the scan instead of accumulating/sorting all windows. Size M.

### Every mutator — including cursor moves and each edit in a batch — re-hashes the entire source PDF   [severity: med] [effort: S]
- FACT (cited): `assert_source_pdf_unchanged` reads annotations.json and MD5s the full source PDF (`_file_md5` at state.py:656, chunked read state.py:620–625). All eight apply.py mutators call it via `_guard_source_pdf` (apply.py:248, 446, 580, 636, 695, 726, 762, 798), including `set_current_annotation` — the status-neutral cursor move that the viewer's Prev/Next buttons land on per its own docstring (apply.py:680–686). `apply_batch` loops `apply_edit` per edit (apply.py:395–397), so N edits ⇒ N full-PDF hashes plus N re-reads of state.json+mapping.json and 2N atomic writes (apply.py:341–342 per call).
- WHY IT MATTERS: For a large source PDF, every Prev/Next click and every item of a Phase-1 batch pays a full-file read+hash; the batch additionally pays O(N) JSON round-trips. Correctness is fine (descending-order application, apply.py:389–391, keeps mappings valid) — this is pure per-operation cost.
- RECOMMENDATION: Cache the guard per (path, mtime, size) within a process, and have `apply_batch` guard once up front then call an internal unguarded apply core; optionally batch the state/mapping writes to one flush at the end. Size S (guard cache) / M (batch write coalescing).

### Extract summary reports the wrong "surfaced_pending" count   [severity: low] [effort: S]
- FACT (cited): `run_extract`'s JSON and human summaries count `surfaced_pending` as `sum(1 for a in annotations if a.trigger_match)` (extract.py:954–956; same expression for the stderr line at extract.py:965). ### No cross-file transactionality: state.json/mapping.json write pairs and commit-then-advance ordering   [severity: low] [effort: M]
- FACT (cited): `apply_edit` and `revert_edit` write state.json then mapping.json as two independent atomic writes (apply.py:341–342; apply.py:529–530) — each file is individually atomic (temp+fsync+rename, state.py:84–92) but a crash between the two leaves an applied status with unshifted mappings. `commit_phase` runs `git commit` first and advances `state["phase"]` afterwards (commit.py:329–352), so (a) the committed state.json snapshot always records the pre-advance phase, and (b) a failure after commit but before the phase write leaves a repo where re-running `commit-phase` with the same `--phase` produces a second commit.
- WHY IT MATTERS: JUDGEMENT: low for a solo, git-backed tool (git history is the recovery path), but worth a deliberate decision before the unified comment model multiplies the number of co-written files.
- RECOMMENDATION: Document the crash-recovery story (one paragraph in state.py's module docstring), and for commit_phase make the re-run idempotent (detect HEAD already matching the rendered message, or advance phase in the same staged write). Size M only if pursuing a journal; the documentation step is S.

### Reusability for the multi-format direction: the core pipeline is closer to format-agnostic than its naming suggests, but the PDF reader drops the data the round-trip needs   [severity: med] [effort: M]
- FACT (cited): (a) `apply_edit`/`revert_edit` treat the target purely as a list of text lines — nothing LaTeX-specific beyond the field name `latex_file` (apply.py:261, 279–300). (b) `fuzzy_map`'s LaTeX coupling is exactly two points: the `_strip_latex` regex pass (extract.py:514–516, applied at :611) and the `"*.tex"` glob (extract.py:530). (c) The status-transition table and validator (state.py:492–598) reference no format at all. (d) `read_annotations` — the engine's only external-comment reader and the matrix's sole "read external comments" path — captures the annotation subtype into a function-local `subtypes_by_id` (extract.py:308, 354) used only for sticky-note association (extract.py:370), and the persisted `Annotation` has no subtype field (field list state.py:191–199); nothing in the fields read at extract.py:317–364 (gettext/contents/author/subtype/boxes/created) captures reply chains (IRT) or annotation ids from the PDF.
- WHY IT MATTERS: The charter's decided direction (full Acrobat-level PDF annotation round-trip incl. sticky notes, StrikeOut, replies; unified comment model) can reuse the transition table, the atomic-write/StateDir layer, the line-edit primitives, and a generalized fuzzy resolver — but the extraction layer as persisted cannot even distinguish a Highlight from a StrikeOut, and reply threads are invisible, so annotations.json schema-v2 is unavoidable for the round-trip.
- RECOMMENDATION: When the schema bumps for the unified model: persist `subtype`, the PDF's native annotation object id, and an optional `in_reply_to` on `Annotation`; rename `latex_file` → `file` at the same time; parameterize `fuzzy_map`'s glob + strip function per format. Size M, and it should ride the same schema-v2 change as the unified comment model rather than land separately.
