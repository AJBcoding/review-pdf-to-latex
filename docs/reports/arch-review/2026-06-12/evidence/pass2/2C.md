### Schema-version guard is bypassed by every mutator and uncaught when it fires   [severity: high] [effort: M]
- FACT (cited): state.py:128 `def read_json` enforces the schema contract (raises
  SchemaVersionError at state.py:149-151, MigrationRequiredError at state.py:159-160).
  But apply.py defines its own unguarded reader at apply.py:121-123 (`_read_json` = bare
  `json.load`) and uses it at apply.py:148, 584, 646-647, 699, 730, 766, 803 — i.e. all
  state.json/mapping.json/annotations.json reads on the mutation path. commit.py uses bare
  `json.load` at commit.py:227 and 282; server.py uses bare `json.loads(...read_text())`
  at server.py:194-195, 429-431, 1111. Only build.py:337, preview.py:186/229, and
  status.py:92 go through the guarded state.read_json. Additionally,
  `grep -n 'SchemaVersionError\|MigrationRequiredError' src/review_pdf_to_latex/cli.py
  tests/test_cli.py` returns NOTHING — no CLI handler catches either exception, and no
  exit code exists for "migration required on read" (the table at cli.py:810-832 has 14
  only for the migrate-state subcommand itself).
- WHY IT MATTERS: the whole migrate.py machinery (migrate.py:14-16: bump schema_version
  last "so a partial migration interrupted by a crash still trips the 'needs migration'
  check on next read") assumes reads are guarded. Today a schema-bumped or
  schema_version-less state.json is silently mutated by apply/set-status/commit-phase, and
  where the guard *does* fire (status/build/preview) it escapes as a raw traceback with
  generic exit 1. This becomes load-bearing the day the unified comment model bumps the
  schema (charter capability direction).
- RECOMMENDATION: (1) delete apply.py:_read_json and route apply/commit/server state reads
  through state.read_json (server's annotations.json read may need the relaxed variant);
  (2) add a top-level catch in cli.main (or per handler) mapping SchemaVersionError /
  MigrationRequiredError to a new spec'd exit code. M overall; the reader swap alone is S.

### Exit-code contract is expressed four different ways, with one numeric overload   [severity: med] [effort: S]
- FACT (cited): the same spec-§8 contract appears as (a) named constants in cli.py:810-832
  (pinned by tests/test_cli.py:1134 `test_exit_code_constants_match_spec`); (b) hardcoded
  class attributes on exceptions — apply.py:33-77 (`exit_code = 7…22`), commit.py:24-50
  (`exit_code = 15…22`); (c) bare magic numbers — extract.py:869 `return 2`, :885
  `return 3`, :904/:911 `return 4`; server.py:1107 `return 5`, :1213 `return 20`; (d) a
  private constants block in pdf_health.py:61-63 where `EXIT_ENCRYPTED = 21` — the same
  number cli.py:830 assigns to `EXIT_SOURCE_PDF_CHANGED` — and run_pdf_health returns it
  as the process exit code (pdf_health.py:259-262). No test cross-checks (b)/(c)/(d)
  against (a): `grep -rn '\.exit_code' tests/test_apply.py tests/test_commit.py` → empty.
- WHY IT MATTERS: the desktop app consumes these as bare numbers
  (one-line out-of-scope note: desktop/main/engine.ts:273 keys success off a
  `successfulExits` numeric set — Pass 3A's problem). Renumbering pressure or a copy-paste
  slip in the unpinned files breaks the engine↔skill↔desktop contract invisibly; exit 21
  already means two different things depending on subcommand.
- RECOMMENDATION: extract a single `exit_codes.py` (or put the table in state.py), import
  it from cli/apply/commit/extract/server/pdf_health, and extend the pinning test to
  assert exception-class exit_codes match the constants. Resolve or explicitly document
  the 21 overload. S.

### Two parallel error-mapping conventions, plus two semantically wrong mappings   [severity: med] [effort: S]
- FACT (cited): convention A — exceptions carry their exit code and handlers do
  `return exc.exit_code` (e.g. cli.py:469-471 for apply; same shape at 497-499, 519-521,
  535-537, 558-560, 584-586, 612-614, 648-650, 705-707). Convention B — bare Exception
  classes mapped manually per class in the handler: _handle_preview at cli.py:668-688
  catches five classes individually because preview.py:158-170 defines its own
  AnnotationNotFoundError / MappingUnresolvedError as plain `Exception` outside the
  ApplyError hierarchy (their docstrings even say "CLI handler maps this to exit code 7/8",
  preview.py:161/167). The class-name duplication itself is 2A/2B territory (flagged in
  pass 1); the split convention is the cross-cutting fact. Additionally two handlers remap
  ValueError to `EXIT_ILLEGAL_STATUS_TRANSITION` (18) for things that are not status
  transitions: cli.py:500-503 (revert: "bad --status / failure-log combos") and
  cli.py:561-564 (append-chat: "Invalid role").
- WHY IT MATTERS: every new subcommand author must pick a convention; preview's manual
  table must be hand-extended for each new failure mode (its handler is already the
  longest, cli.py:655-690). The ValueError→18 mappings report a misleading error category
  to the skill/desktop.
- RECOMMENDATION: fold preview's exceptions into the exit-code-carrying hierarchy
  (a shared base in state.py or the proposed exit_codes module) so all handlers collapse
  to the `return exc.exit_code` shape; give the two ValueError cases honest codes (9 for
  unreadable failure-log, or a generic usage-error code). S.

### `--json` is a global flag that most subcommands silently ignore; errors are never JSON   [severity: low] [effort: S]
- FACT (cited): `--json` is attached to the top-level parser and every subparser
  (cli.py:76-82, help text "where supported"). Only three handlers read it: extract
  (cli.py:304), status (cli.py:443), bulk-surface (cli.py:709); pdf-health is always-JSON
  by design (cli.py:285-286). apply/revert/set-status/set-current/append-chat/
  record-proposal/override-mapping/commit-phase/preview never reference `json_output`
  (verified across cli.py:450-690 — no occurrence; full-file grep at cli.py shows
  occurrences only at lines 304, 443, 709, plus the definition). All error paths print
  free-text to stderr (e.g. cli.py:470 `print(f"error: {exc}", file=sys.stderr)`).
- WHY IT MATTERS: the Electron app and the skill drive this CLI programmatically; today
  they get machine output from 4 of 17 subcommands and must parse prose stderr for
  failures. The unified-model direction will add more programmatic callers.
- RECOMMENDATION: either honor `--json` uniformly (success envelope + structured error
  envelope `{error, exit_code}` on stderr) or scope the flag to the subcommands that
  support it so `apply --json` errors loudly instead of lying. S–M depending on choice; a
  minimal S step is the structured error envelope.

## Reusability for the multi-format direction (JUDGEMENT, grounded in the above)

The CLI router is already format-agnostic at the dispatch layer: only `extract --pdf`
(cli.py:109) and build's LaTeX engines (cli.py:166-169) are format-coupled; every
state-mutation subcommand (set-status, append-chat, record-proposal, override-mapping,
set-current) operates on annotation IDs and would carry over to a unified comment model
unchanged. The exit-code contract + `--json` envelope is the natural engine↔desktop seam —
which is why findings 2 (schema guard), 3 (exit-code single source) and 5 (uniform JSON)
are the cheapest pre-work for that direction. apply_batch (apply.py:353) is a ready-made
bulk-apply primitive if the unified model needs one.
